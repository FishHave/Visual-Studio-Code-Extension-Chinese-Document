# 测试API

测试API让Visual Studio Code扩展得以探索工作空间测试，并发布测试结果。用户可以在测试浏览器视图、修饰符和内部命令执行测试。有了这些新的API，Visual Studio Code支持更丰富的输出与差异显示，创造更多的可能。

> **注意**： 测试API仅对VS Code 1.59以上版本可用。

## 示例

已经由两种由VS Code团队维护的测试提供者：

- [示例测试扩展](https://github.com/microsoft/vscode-extension-samples/tree/main/test-provider-sample)，提供Markdown文件测试。

- [自托管测试扩展](https://github.com/microsoft/vscode-selfhost-test-provider)，我们用于运行VS Code自身的测试。

## 发现测试

测试由`TestController`提供，它需要全局唯一ID和人类可读标签创建测试：

```typescript
const controller = vscode.tests.createTestController(
    'helloWorldTests',
    'Hello World Tests'
);
```

如需发布测试，添加`TestItem`作为控制器`items`集合的子元素。`TestItem`接口中的`TestItem`是测试API的基础，是描述存在于代码中的测试用例、集或树项目的通用类型。它们可以各自拥有`children`，形成一个层次结构。例如，下面是样例测试扩展创建测试的简化版本：

```typescript
parseMarkdown(content, {
    onTest: (range, numberA, mathOperator, numberB, expectedValue) => {
        // 如果这是最高等测试，那么添加至父的子对象。如果不是
        // 添加至控制器的最高等项目。
        const collection = parent ? parent.children : controller.items;
        // 创建新的ID，在父的子对象中唯一：
        const id = [numberA, mathOperator, numberB, expectedValue].join('  ');

        // 最后， 创建测试项
        const test = controller.createTestItem(id, data.getLabel(), item.uri);
        test.range = range;
        collection.add(test);
    }
    //...
});
```

类似于诊断，如何发现测试通常取决于扩展。简单的扩展可能会观测整个工作空间，并解析所有文件中可用的测试。然而，立即解析所有对于大工作空间来说会很慢。因此，您可以做两件事：

1. 通过监测`vscode.workspace.onDidOpenTextDocument`，在编辑器打开文件时主动发现测试。

2. 设置`item.canResolveChildren = true`，并设置`controller.resolveHandler`。`resolveHandler`在用户操作请求发现测试时调用，比如在测试浏览器中展开项目。

下面是在惰性解析文件的扩展中，此策略的样貌：

```typescript
// 受限，创建`resolveHandler`。这会被立刻用“undefined”调用
// 来在工作空间中请求所有被发现的测试，通常，
// 发生在用户第一次打开测试浏览器时。
controller.resolveHandler = async test => {
    if (!test) {
        await discoverAllFilesInWorkspace();
    } else {
        await parseTestInFileContents(test);
    }
};

// 当打开文本文档时，解析其中的测试。
vscode.workspace.onDidOpenTextDocument(parseTestsInDocument);
// 我们也能监听文档变化，以重新解析未保存的更改：
vscode.workspace.onDidChangeTextDocument(e => parseTestInDocument(e.document));

// 在本函数中，如果我们已经发现了文件的`TestItem`， 我们就会获取它，
// 否则，我们会用`canResolveChildren = true`创建它，表明它
// 可以传递给`controller.resolveHandler`以获取其子对象。
function getOrCreateFile(uri: vscode.Uri) {
    const existing = controller.items.get(uri.toString());
    if (existing) {
        return exsiting;
    }

    const file = controller.createTestItem(uri.toString(), uri.path.split('/').pop()!, uri);
    file.canResolveChildren = true;
    return file;
}

function parseTestsInDocument(e: vscode.TextDocument) {
    if (e.uri.scheme === 'file' && e.uri.path.endsWith('.md')) {
        parseTestsInFileContents(getOrCreateFile(e.uri), e.getText());
    }
}

async function parseTestsInFileContents(file: vscode.TestItem, contents?: string) {
    // 如果文件被打开，VS Code已经知道了它的内容。如果
    // resolveHandler在文档未被打开时调用了此函数，我们需要
    // 自己从磁盘读取。
    if (contents === undefined) {
        const rawContent = await vscode.workspace.fs.readFile(file.uri);
        contents = new TextDecoder().decode(rawContent);
    }

    // 一些自定义逻辑，以填充来自内容的test.children
}
```

`discoverAllFilesInworkspace`的实现可以使用VS Code已有的文件监测功能构建。当调用`resolveHandler`时，您应当继续监测更改，以便测试浏览器保持数据更新。

```typescript
async function discoverAllFilesInWorkspace() {
    if (!vscode.workspace.workspaceFolders) {
        return []; // 处理无文件夹打开的情况。
    }

    return Promise.all(
        vscode.workspace.workspaceFolders.map(async workspaceFolder => {
            const pattern = new vscode.RelativePattern(workspaceFolder, '**/*.md');
            const watcher = vscode.workspace.createFileSystemWatcher(pattern);

            // 创建文件时，确保树中有关联的“file”节点
            watcher.onDidCreate(uri => getOrCreateFile(uri));
            // 当文件改动时，重新解析。注意您可以优化这个过程
            // 以便您只需要重新解析在过去被处理过的子对象。
            watcher.onDidChange(uri => parseTestInFileContents(getOrCreateFile(uri)));
            // 最后，为已经移除的文件删除TestItem。因为
            // 我们使用URI作为TestItem的ID，这做起来很容易。
            watcher.onDidDelete(uri => controller.items.delete(uri.toString()));

            for (const file of await vscode.workspace.findFiles(pattern)) {
                getOrCreateFile(file);
            }

            return watcher;
        })
    );
}
```
`TestItem`接口十分简单，并不需要给自定义数据留出空间。如果您需要关联额外信息，可以使用[`WeakMap`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)：

```typescript
const testData = new WeakMap<vscode.TestItem, MyCustomData>();

// 关联数据：
const item = controller.createTestItem(id, label);
testData.set(item, myCustomData());

// 之后拿回来：
const myData = testData.get(item);
```

保证传递给所有`TestController`相关方法的`TestItem`实例与自`createTestItem`创建的是同一个，这样您可以确保从`testData`映射获取的项目正常工作。

在本案例中，让我们只存储每个项目的类型：

```typescript
enum ItemType {
    File,
    TestCase
}

const testData = myWeakMap<vscode.TestItem, ItemType>();

const getType = (testItem: vscode.TestItem) => testData.get(testItem)
```

## 运行测试
测试通过`TestRunProfile`运行。每份资料属于具体的执行`kind`（类型）：运行、调试或者是覆盖率。多数测试扩展在各自的组中最多有一份资料，但多份是允许的。例如，如果您的扩展在多个平台运行测试，您可以为每个平台绑定一份资料和`kind`（类型）。每个资料拥有一个`runHandler`，在该类型的运行收到请求时调用。

```typescript
function runHandler(
    shouldDebug: boolean,
    request: vscode.TestRunRequest,
    token: vscode.CancellationToken
) {
    // todo
}

const runProfile = controller.createRunProfile(
    'Run',
    vscode.TestRunProfileKind.Run,
    (request, token) => {
        runHandler(false, request, token);
    }
);

const debugProfile = controller.createRunProfile(
    'Debug',
    vscode.TestRunProfileKind.Debug,
    (request, token) => {
        runHandler(true, request, token);
    }
);
```

`runHandler`至少应当调用`controller.createTestRun`一次，将原请求传递过去。请求包含在测试运行中`include`（涵盖）的测试（如果用户要求运行所有测试将被忽略），可能会包含运行`exclude`（之外）的测试。扩展应当使用产生的`TestRun`对象更新运行涉及的测试状态。例如：

```typescript
async function runHandler(
    shouldDebug:boolean,
    request: vscode.TestRunRequest,
    token: vscode.CancellationToken
) {
    const run = controller.createTestRun(request);
    const queue: vscode.TestItem[] = [];

    // 遍历所有包含的或已知的测试，将其入队
    if (request.include) {
        request.include.forEach(test => queue.push(test));
    } else {
        controller.items.forEach(test => queue.push(test));
    }

    // 尝试运行每个入队的测试。调用run.passed()或者run.failed()。
    // `TestMessage`可以容纳额外信息，比如失败位置或
    // 差异输出。但是在此我们只输入文本消息。
    while (queue.length > 0 && !token.isCancellationRequested) {
        const test = queue.pop()!;

        // 当用户要求排除时跳过测试
        if(request.exclude?.includes(test)) {
            continue;
        }

        switch (getType(test)) {
            case ItemType.File:
                // 如果我们正在运行文件，并且尚未知道它里面有什么，现在解析它
                if (test.children.size === 0) {
                    await parseTestsInFileContents(test);
                }
                break;
            case ItemType.TestCase:
                // 否则，运行测试用例。注意我们不需要手动
                // 设置父测试的状态；它们会自动设置
                const start = Date.now();
                try {
                    await assertTestPasses(test);
                    run.passed(test, Date.now() - start);
                } catch (e) {
                    run.failed(test, new vscode.TextMessages(e.message), Date.now() - start);
                }
                break;
        }

        test.children.forEach(test => queue.push(test));
    }

    // 确保所有测试执行后结束运行：
    run.end();
}
```

除了`runHandler`之外，您还可以设置`TestRunProfile`中的`configureHandler`。如果有，VS Code会提供UI，让用户配置测试运行，并在他们如此做后调用处理器。从此，您可以打开文件，显示快速选单，或者做任何适合您的测试框架的事情。

> VS Code有意将测试配置和调试或者任务配置区分开。这些是传统编辑器或IDE核心特性，在`.vscode`文件夹中的特殊文件配置。然而，传统测试从命令行执行，大多数测试框架有现成的配置策略。因此，在VS Code，我们避免了配置的赋值，而是将其留给扩展处理。

### 测试输出
除了传递消息给`TestRun.failed`或者`TestRun.errored`，您可以使用`run.appendOutput(str)`加入通用的输出。这个输出可以在命令行解释器中使用**测试：显示输出**，通过各种UI按钮显示，比如测试浏览器视图的命令行解释器图标。

由于字符串在命令行解释器中渲染，您可以使用全部[ANSI编码](https://en.wikipedia.org/wiki/ANSI_escape_code)，包括在[ansi-styles](https://www.npmjs.com/package/ansi-styles)npm包中可用的样式。记住，因为它处在命令行解释器中，行必须使用CRLF（`\r\n`）而只是LF，通常作为一些工具的默认输出，来（`\n`）包裹。

### 测试覆盖率

测试覆盖率通过`run.addCoverage()`方法与`TestRun`关联。按规范，这应该通过`TestRunProfileKind.Coverage`资料的`runHandler`完成，但在任何测试运行中调用其是可能的。`addCoverage`方法接收`FileCoverage`对象，作为文件中的覆盖率数据摘要：

```typescript
async function runHandler(
    shouldDebug: boolean,
    request: vscode.TestRunRequest,
    token: vscode.CancellationToken
) {
    // ...
    for await (const file of readCoverageOutput()) {
        run.addCoverage(new vscode.FileCoverage(file.uri, file.statementCoverage));
    }
}
```

`FileCoverage`包含每个文件中所有语句、分支和声明的覆盖或未覆盖的数目。基于您的运行时和覆盖率格式，您可能看到作为行覆盖率引用的语句覆盖率，或者作为函数或方法覆盖率引用的声明覆盖率。您可以添加文件覆盖率，实现单个URI，多次覆盖，在此情况下新信息会替换旧信息。

用户打开具有覆盖率的文件，或在**测试覆盖率**视图中展开文件，VS Code请求关于此文件的更多信息。它通过使用`TestRun`、`FileCoverage`和`CancellationToken`，在`TestRunProfile`上调用扩展定义的`loadDetailedCoverage`方法来完成。注意测试运行和文件覆盖率实例与在`run.addCoverage`中使用的相同，这对关联数据大有帮助。例如，您可以创建`FileCoverage`对象和您自己数据的映射：

```typescript
const coverageData = new WeakMap<vscode.FileCoverage, MyCoverageDetails>();

profile.loadDetailedCoverage = (testRun, fileCoverage, token) => {
    return coverageData.get(fileCoverage).load(token);
};

async function runHandler(
    shouldDebug: boolean,
    request: vscode.TestRunRequest,
    token: vscode.CancellationToken
) {
    // ...
    
    for await (const file of readCoverageOutput()) {
        const coverage = new vscode.FileCoverage(file.uri, file.statementCoverage);
        coverageData.set(coverage, file);
        run.addCoverage(coverage);
    }
}
```

您可能还能实现包含那个数据的子类`FileCoverage`：

```typescript
class MyFileCoverage extends vscode.FileCoverage {
    // ...
}

profile.loadDetailedCoverage = async (testRun, fileCoverage, token) => {
    return fileCoverage instanceof MyFileCoverage ? await fileCoverage.load() : [];
};

async function runHandler(
    shouldDebug: boolean,
    request: vscode.TestRunRequest,
    token: vscode.CancellationToken
) {
    // ...

    for await (const file of readCoverageOutput()) {
        // `file`是MyFileCoverage
        run.addCoverage(file);
    }
}
```
