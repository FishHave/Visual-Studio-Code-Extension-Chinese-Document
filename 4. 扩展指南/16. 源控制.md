# 源控制

源控制API让扩展作者能够定义源控制管理（SCM）特性。Visual Studio Code中存在一个轻量但强大的API接口，能够整合许多不同的SCM系统，同时提供一个共通的用户接口。

![](https://code.visualstudio.com/assets/api/extension-guides/scm-provider/main.png)

VS Code本身搭载了一个源控制提供者：Git扩展，它是这个API最好的注脚，对于想要贡献自己专属的SCM提供者的人来说也是一个[不错的起点](https://github.com/microsoft/vscode/blob/main/extensions/git/src/repository.ts)。在市场中有其他优秀的案例，比如[SVN扩展](https://marketplace.visualstudio.com/items?itemName=johnstoncode.svn-scm)。

> 注意：您可以随时在本文档中查看[`vscode`命名空间API参考](/9.%20参考/1.%20VS%20Code%20API.md)

# 源控制模型

`SourceControl`是一种实体，它负责写入源控制模型的**资源状态**，也就是`SourceControlResourceState`的实例。资源状态本身以组的方式阻止，组是`SourceControlResourceGroup`的实例。

您可以用`vscode.scm.createSourceControl`创建新的SourceControl。

为了更好地理解这三个相互关联的实体，让我们以[Git](https://github.com/microsoft/vscode/tree/main/extensions/git)为例。请看下面`git status`的输出：

```bash
vsce main* → git status
On branch main
Your branch is up-to-date with 'origin/main'.
Changes to be committed:
  (use "git reset HEAD <file>..." to unstage)

        modified:   README.md
        renamed:    src/api.ts -> src/test/api.ts

Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

        deleted:    .travis.yml
        modified:   README.md

```

这个工作空间正发生翻天覆地的变化。受限`README.md`文件被修改、暂存、再次修改。然后，`src/api.ts`文件被移动到`src/test/api.ts`，移动随后被暂存。最后，`.travis.yml`文件被删除。

对于此工作空间，Git定义两个资源组：**工作树**和**索引**。组内的**文件更改**就是**资源状态**：
- **索引** - 资源组
    - `README.md`，修改 - 资源状态
    - `src/test/api.ts`，重命名自`src/api.ts` - 资源状态
- **工作树** - 资源组
    - `.travis.yml`, 删除 - 资源状态
    - `README.md`，修改 - 资源状态

注意同一个文件`README.md`是两个不同的资源状态的一部分。

下面，Git创建此模型：

```typescript
function createResourceUri(relativePath: string) : vscode.Uri {
    const absolutePath = path.join(vscode.workspace.rootPath, relativePath);
    return vscode.Uri.file(absolutePath);
}

const gitSCM = vscode.scm.createSourceControl('git', 'Git');

const index = gitSCM.createResourceGroup('index', 'Index');
index.resourceStates = [
    { resourceUri: createResourceUri('README.md') },
    { resourceUri: createResourceUri('src/test/api.ts') }
];

const workingTree = gitSCM.createResourceGroup('workingTree', 'Changes');
workingTree.resourceStates = [
    { resourceUri: createResourceUri('.travis.yml') },
    { resourceUri: createResourceUri('README.md') }
];
```

对源控制和资源组的修改会传递到源控制视图。

## 源控制视图
VS Code能在源控制模型发生更改时对源控制视图输入数据。资源状态可以使用`SourceControlResourceDecorations`自定义：
```typescript
export interface SourceControlResourceState {
    readonly decorations?: SourceControlResourceDecorations;
}
```

上一个案例大概足够在源控制视图中填充一个简单的列表，但如果用户期望对各个资源交互。例如，当用户点击资源状态会发生什么？资源状态可以提供一条命令解决这个问题：

```typescript
export interface SourceControlResourceState {
    readonly command?: Command;
}
```

### 菜单
您可以在六个源控制菜单ID中放置菜单项，以便给用户提供丰富的用户界面。

`scm/title`菜单位于SCM视图标题的有房。`navigation`组的菜单项会在行间，其他的都在`...`下拉菜单。

这三个具有相似之处：
- `scm/resourceGroup/context`添加命令到[`SourceControlResourceGroup`](/9.%20参考/1.%20VS%20Code%20API.md)项。
- `scm/resourceState/context`添加命令到[`SourceControlResourceState`]项。
- `scm/resourceFolder/context`添加命令到居中的文件夹。当[`SourceControlResourceState`](/9.%20参考/1.%20VS%20Code%20API.md)的resourceUri路径包含文件夹，且用户选择了树视图而非列表视图模式时该文件夹会出现。

在`inline`组放置菜单项使之处于行间。所有其他菜单项组会在上下文菜单中出现，通常可以使用鼠标右键访问。

注意SCM视图支持多选，所以命令接收的参数为一个或多个资源的数组。

例如：Git支持通过添加`git.stage`命令到`scm/resourceState/context`菜单暂存多个文件，使用下面的声明：

```typescript
stage(...resourceStates: SourceControlResourceState[]) : Promise<void>;
```

当被创建时，`SourceControl`和`SourceControlResourceGroup`实例需要您提供`id`字符串。这些值会分别填入`scmProvider`和`scmResourceGroup`上下文键。您可以在菜单项目的时句中利用这些[上下文键](/9.%20参考/6.%20时句上下文.md)。下面是Git对`git.stage`命令展示内联菜单项目的方法：

```json
{
    "command": "git.stage",
    "when": "scmProvider == git && scmResourceGroup == merge",
    "group": "inline"
}
```

`scm/repository`菜单是**源控制仓库**视图中每个`SourceControl`实例的菜单。在`inline`组中放置菜单项使它们能在行间出现。所有其他菜单项组会在`...`菜单中显示。`inline`组会结合可用的空间渲染，不能自动适应的菜单项会自动移入`...`菜单。

`scm/sourceControl`菜单在**源控制仓库**视图中，是每个`SourceControl`实例的上下文菜单：

![](https://code.visualstudio.com/assets/api/extension-guides/scm-provider/sourcecontrol-menu.png)

在此[提前](#快速差异对比)介绍，`scm/change/title`允许贡献命令到[快速差异对比](/9.%20参考/1.%20VS%20Code%20API.md)的内联差异编辑器的标题栏。命令需带参数传递：文档URI和更改数组，以及内联更改差异编辑器正在负责的更改索引。例如，下面是`stageChange`Git命令的声明，它被贡献到此菜单，并附有时句检测`originalResourceScheme`[上下文键](/9.%20参考/6.%20时句上下文.md)是否等同于`git`：

```typescript
async stageChange(uri: Uri, changes: LineChange[], index: number): Promise<void>;
```

### SCM输入框

源控制输入框，位于每个源控制视图的上方，允许用户输入信息。您可以获取（和设置）此信息来进行一些操作。例如，在Git中，这用作提交框，用户在其中键入提交信息，然后`git commit`命令使用之。

```typescript
export interface SourceControlInputBox {
    value: string
}

export interface SourceControl {
    readonly inputBox: SourceControInputBox;
}
```

用户可以按下<kbd>Ctrl+Enter</kbd>（macOS下为<kbd>Cmd+Enter</kbd>）接受任何消息。您可以提供`acceptInputCommand`到`SourceControl`实例以处理此事件。

```typescript
export interface SourceControl {
    readonly acceptInputCommand?: Command;
}
```

## 快速差异对比
VS Code支持显示**快速差异对比**编辑器侧边栏修饰。点击修饰会触发内联差异体验，在此您可以贡献上下文命令。
![](https://code.visualstudio.com/assets/api/extension-guides/scm-provider/quickdiff.png)

这些修饰由VS Code本身计算。您需要做的所有工作是给VS Code提供任意给定文件的原始内容。

```typescript
export interface SourceControl {
    quickDiffProvider?: QuickDiffProvider;
}
```

使用`QuickDiffProvider`的`provideOriginalResource`方法，您的实现可以告知VS Code匹配`uri`作为方法参数的资源的原始资源之`Uri`。

结合此API和[`workspace`命名空间中的`registerTextDocumentContentProvider`方法](/9.%20参考/1.%20VS%20Code%20API.md)让您提供任意资源的内容，只要给定已注册`scheme`的`Uri`。
