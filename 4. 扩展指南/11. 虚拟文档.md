# 虚拟文档

文本文档内容提供者API允许您在Visual Studio Code创建来自任意源的只读文档。您可以在 https://github.com/microsoft/vscode-extension-samples/blob/main/virtual-document-sample/README.md 找到附有源代码的实例扩展。

## TextDocumentContentProvider

API通过声明URI框架来运作。您的提供者将以此框架返回文本内容。这种框架必须在注册提供者时提供，并且在之后无法更改。相同的提供者可以由多个框架使用，多个提供者可以注册为同个框架。

```typescript
vscode.workspace.registerTextDocumentContentProvider(myScheme, myProvider);
```

调用`registerTextDocumentContentProvider`返回一个一次性对象，凭此对象注册可以被撤销。任意提供者必须仅实现`provideTextDocumentContent`函数，此函数与URI和取消令牌一同调用。
```typescript
const myProvider = new (class implements vscode.TextDocumentContentProvider {
    provideTextDocumentContent(uri: vscode.Uri): string {
        // 使用URI路径作为文本调用Cowsay
        return cowsay.say({ text: uri.path });
    }
})();
```

请注意提供者不会为虚拟文档创建URI——它的作用是**提供**具有此URI的内容。在返回中，内容提供者被接入开放的文档逻辑，这样提供者永远会被注意到。

本案例使用“cowsay”——一条创建编辑器之后显示的URI的命令。

```typescript
vscode.commands.registerCommand('cowsay.say', async () => {
    let what = await vscode.window.showInputBox({ placeHolder: 'cow say?'});
    if (what) {
        let uri = vsocde.Uri.parse('cowsay:' + what);
        let doc = await vscode.workspace.openTextDocument(uri); // 回调提供者
        await vscode.window.showTextDocument(doc, { preview: false });
    }
});
```

本命令提示输入，创建`cowsay-`框架URI，打开URI的文档，最后用编辑器打开文档。在第三步，打开文档时，询问提供者提供URI的内容。

拥有此命令，我们有了具有完整功能的文本文档内容提供者。下面的部分介绍虚拟文档如何更新，虚拟文档的UI命令如何注册。

### 注册虚拟文档
基于场景，虚拟文档可能更改。为了支持这种场景，提供者可以实现`onDidChange`事件。

`vscode.Event`类型定义了VS Code事件协议。实现事件最简单的方法是`vscode.EventEmitter`，如此：
```typescript
const myProvider = new (class implements vsocde.TextDocumentContentProvider {
    // 发送者和其命令
    onDidChangeEmitter = new vscode.Emitter<vscode.Uri>();
    onDidChange = this.onDidChangeEmitter.event;

    //...
})();
```

事件发送者具有`fire`方法，用于提醒VS Code有个更改在文档发生。被更改文档由其URI识别，URI会作为参数传递给`fire`方法。假定文档仍然打开的话，提供者随后再次被调用提供更新后的内容。

所有使VS Code监听虚拟文档更改的事情都已经做好了。为了查看更复杂的实例，充分利用其特性，看看：https://github.com/microsoft/vscode-extension-samples/blob/main/contentprovider-sample/README.md 。
### 添加编辑器命令

可以添加仅与关联内容提供者提供的文档互动的编辑器操作。这是一个实例命令，反转cow刚刚所说的话：
```typescript
// 注册命令更新当前的cowsay
subscriptions.push(
    vscode.commands.registerCommand('cowsay.backwards', async () => {
        if(!vsocde.window.activeTextEditor) {
            return; // 无编辑器
        }
        let { document } = vscode.window.activeTextEditor;
        if(document.uri.scheme !== myScheme) {
            return; // 不是我的框架
        }
        // 获取路径组件，反转之，创建新URI
        let say = document.uri.path;
        let newSay = say
            .split('')
            .reverse()
            .join('');
        let newUri = document.uri.with({ paht: newSay });
        await vscode.window.showTextDocument(newUri, { preview: false });
    })
);
```
上面的片段检查我们是否有活动编辑器，和其文档是否是我们的框架。这些检查是必要的，因为命令对所有文档可用（和可执行）。随后URI路径组件会被反转，并从之创建新URI，之后，打开一个编辑器。

编辑器命令的首要之事是需要`package.json`的声明部分。在`contributes`部分添加下面的配置信息：
```json
"menus": {
    "editor/title": [
        {
            "command": "cowsay.backwards",
            "group": "navigation",
            "when": "resourceScheme == cowsay"
        }
    ]
}
```
这里指向在`contributes/commands`部分定义的`cowsay.backwards`命令，并且声明它需要在编辑器标题栏出现（右上方的工具栏）。现在，到此为止，命令会对所有编辑器一直显示。这就是时词发挥作用的时候——它声明什么条件下显示操作。在本例，它说明编辑器中文档的框架必须为`cowsay-`框架。在`commandPalette`菜单中重复此配置信息，它默认显示所有命令。

### 事件与可见性

文档提供者是VS Code中的“一等公民”，它们的内容在常规文本文档中出现，它们使用文件相同的底层等等。然而，这同样意味着“您的”文档无法隐藏，它们会在`onDidOpenTextDocument`和`onDidCloseTextDocument`事件中出现，它们是`vscode.workspace.textDocuments`的一部分等等。所有人应该遵守检查文档`scheme`和确定您是否想要对此文档做一些事情这样的规则。

## 文件系统API

如果您需要更多的灵活性和能力，看看[`FileSystemProvider`](/9.%20参考/1.%20VS%20Code%20API.md)API。它允许实现完整的文件系统，具有文件、文件夹、二进制数据、文件检测、创建和更多。

您可以在 https://github.com/microsoft/vscode-extension-samples/tree/main/fsprovider-sample/README.md 找到具有源码的示例扩展。

当VS Code打开这样的文件系统中的文件夹或工作空间时，我们称之为虚拟工作空间。当VS Code窗口中打开虚拟工作空间时，左下方的远程标志会显示一个标签，类似于远程窗口。参考[虚拟工作空间指南](/4.%20扩展指南/12.%20虚拟工作空间.md)了解扩展如何支持这样的初始化。
