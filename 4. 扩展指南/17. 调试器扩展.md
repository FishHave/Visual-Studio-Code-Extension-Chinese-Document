# 调试器扩展

Visual Studio Code的调试器构造允许扩展作者简易地整合现有的调试器到VS Code中，并为其实现通用的用户界面。

VS Code搭载了一个内建调试器扩展，[Node.js](https://nodejs.org/)调试器扩展，可以作为VS Code支持的许多调试器功能的最佳演示。

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/debug-features.png)

此截图展示了下面的调试器特性：

1. 调试器配置管理。
2. 开始/停止和步入等调试操作。
3. 源、函数、条件、内联断点及日志点。
4. 栈追踪，包括多线程和多进程支持。
5. 视图和悬停中的复杂数据结构浏览
6. 悬停或源内联中的变量值显示
7. 管理表达式监控
8. 提供带有自动补全的交互预测的调试控制台

本文档将帮助您创建调试器扩展，可以实现所有VS Code中调试器的工作。

## VS Code中的调试器构造

我们已经引入了一个抽象协议与调试器后端交流，VS Code使用这个协议实现了一个通用（未知语言下）调试器UI。因为调试器通常不实现此协议，所以需要一些中介来“适配”调试器到该协议。中介一般是单独的进程，与调试器进行交流。

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/debug-arch1.png)

我们叫这个中介**调试适配器**（简称**DA**），用在DA和VS Code之间的抽象协议叫**调试适配器协议**（简称**DAP**）。由于调试适配器协议与VS Code相互独立，所以它有自己的[网站](https://microsoft.github.io/debug-adapter-protocol/)，您可以在那找到一份[介绍和概览](https://microsoft.github.io/debug-adapter-protocol/overview)，详细的[说明](https://microsoft.github.io/debug-adapter-protocol/specification)和一些[著名的实现和支持工具]列表(https://microsoft.github.io/debug-adapter-protocol/implementors/adapters/)。DAP背后的历史和启发在这篇[博客帖子](https://code.visualstudio.com/blogs/2018/08/07/debug-adapter-protocol-website#_why-the-need-for-decoupling-with-protocols)有所解释。

既然扩展适配器自VS Code独立，也可以用在[其他开发工具](https://microsoft.github.io/debug-adapter-protocol/implementors/tools/)，那么它们就不需要适配VS Code基于扩展和贡献点的扩展构造。

出于这个原因，VS Code提供了一个贡献点：`debuggers`，在这您可以指定一个调试类型（例如`node`对于Node.js调试器）来贡献调试适配器。无论用户在什么时候开始此类型的调试会话，VS Code都会启动在这注册的DA。

所以在它最简洁的形式中，调试器扩展仅仅只是一个声明性的调试适配器实现贡献，扩展只是一个基本的调试适配器包装，而不带有任何额外代码。

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/debug-arch2.png)

更实际的调试器扩展贡献了VS Code中许多或全部下面的声明性项目：
1. 一系列由调试器支持的语言。VS Code允许UI为那些语言设置断点。
2. 调试器引入的调试器配置属性专属JSON框架。VS Code使用此框架验证launch.json编辑器中的属性并提供智能感知。请注意JSON框架中的`$ref`和`definition`部分不受支持。

您可以在[`contributes.breakpoints`](/9.%20参考/2.%20贡献点.md)和[`contributes.debuggers`](/9.%20参考/2.%20贡献点.md)参考找到更多信息。

除了上述完全的声明性贡献，调试扩展API开放了基于代码的功能：

- 为VS Code创建的初始launch.json动态构建默认扩展配置。
- 控制调试适配器的动态使用。
- 在传递给调试适配器前验证或修改调试配置信息。
- 与调试适配器互通。
- 发送消息给调试控制台。

在本文档的剩余部分，我们展示了如何开发调试器扩展。

# 演示调试扩展

因为从零开始创建调试配置器对于这个教程来说有点小题大做，所以我们使用教学用“调试配置器新手工具包”中创建的简易DA入门。它之所以叫做*演示调试*，是因为我们不会讨论真正的调试器，而是仿制的那种。演示调试模拟了调试器，支持步入、继续、断点、异常和变量访问，但它与任何真正的调试器没有关系。

在探究演示调试的开发步骤前，我们先安装VS Code市场中的[预构建版本](https://marketplace.visualstudio.com/items?itemName=ms-vscode.mock-debug)，并与其进行尝试：

- 切换到扩展视图，选择类型“mock”检索Mock Debug Extension，
- “安装”并“重新加载”扩展。

尝试演示调试：
- 创建新的空文件夹`mock test`，在VS Code中打开它。
- 创建文件`readme.md`，随便输入几行文本。
- 切换到运行和调试视图（<kbd>Ctr+Shift+D</kbd>），选择**创建launch.json文件**链接。
- VS Code会让您选择一个调试器，以便创建默认的启动配置。选择“Mock Debug”。
- 按下绿色的**开始**按钮，然后键入<kbd>Enter</kbd>确认建议的文件`readme.md`。

将启动调试会话，您可以“进入”readme文件，设置并触发断点，进入异常（如果某行中有出现`exception`这个单词）。

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/mock-debug.gif)

在使用演示调试作为您自己的开发的起点前，我们建议先卸载预构建的版本：

- 切换到扩展视图，点击演示调试扩展的剪刀图标。
- 使用“卸载”操作，随后“重新加载”窗口。

## 演示调试的开发初始化
现在，让我们获取演示调试的源代码，在VS Code开始对它的开发。

```bash
git clone https://github.com/microsoft/vscode-mock-debug.git
cd vscode-mock-debug
yarn
```

在VS Code中打开项目文件夹`vscode-mock-debug`。

包里有什么？
- `package.json`是演示调试扩展的清单：
    - 它列出了演示调试扩展的贡献。
    - `compile`和`watch`脚本用于转译TypeScript源代码到`out`文件夹，并监视随后的源代码修改。
    - 依赖`vscode-debugprotocol`、`vscode-debugadapter`和`vscode-debugadapter-testsupport`s NPM模块，用来简化基于node的调试适配器开发。
- `src/mockRuntime.ts`是*演示*运行时，附有简单的调试API
- 适配运行时到调试适配器协议生态的代码位于`src/mockDebug.ts`。在此您可以找到DAP不同请求的处理器。
- 因为调试器扩展的实现基于调试适配器，我们不需要让扩展拥有代码（即它运行在扩展宿主进程）。然而，演示调试有小小的`src/extension.ts`，因为它展示了扩展代码中可以对调试器扩展做什么。

现在选择**扩展**启动配置，按下`F5`，构建并启动演示扩展。最初，这需要完全转译TypeScript源代码到`out`文件夹。在完全构建后，一个*监视器任务*将启动，监控您做出的任何修改。

转译源代码后，标签为“[扩展开发宿主]”的全新VS Code窗口会与演示调试扩展一同出现，并运行在调试模式。在此窗口打开带有`readme.md`文件的`mock test`项目，用“F5”启动调试会话，并进入：

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/debug-mock-session.png)

由于您正在调试模式下运行扩展，您可以设置并触发`src/extension.ts`中的断点，但如我们上面提及的，扩展中并没有什么有趣的代码在执行。有点意思的代码都运行在调试适配器，而它在一个独立的进程中。

为了调试调试适配器本身，我们需要在调试模式下运行它。这可以通过在*服务端模式*下运行调试适配器，并配置VS Code连接来做到。在VS Code vscode-mock-debug项目，从下拉菜单中选择启动配置**服务端**，按下绿色的开始按钮。

既然我们已经有一个供扩展使用且活跃的调试会话，VS Code调试器UI现在会进入*多会话*模式，这可以通过在栈调用中看到两个调试会话的名字：**扩展**和**服务端**判断：

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/debugger-extension-server.png)

现在我们可以同时调试扩展和模拟的DA。到达这里更快的方法是使用**扩展+服务端**启动配置，会自动启动两个会话。

[下面](开发调试器扩展的额外途径)可以找到另一个更简单的调试扩展和DA的方式选择。

在`src/mockDebug.ts`中的方法`launchRequest()`的开头设置断点，最后一步配置，为您的演示测试启动配置添加`debugServer`属性为端口`4711`让演示调试器连接DA服务端：

```json
{
    "version": "0.2.0",
    "configuration": [
        {
            "type": "mock",
            "request": "launch",
            "name": "mock test",
            "program": "${workspaceFolder}/readme.md",
            "stopOnEntry": true,
            "debugServer": 4711
        }
    ]
}
```

如果您现在启动此调试配置，VS Code不会启动演示调试适配器作为单独的进程，而是会直接连接到已运行服务端的本地端口4711，所以您会触发`launchRequest`的断点。

在这个初始化设置之后，您可以轻便地修改、转译并调试演示调试。

但是现在真正的问题显现：您会不得不用一些和“真正的”调试器或运行时的代码替换`src/mockDebug.ts`和`src/mockRuntime.ts`中的调试适配器演示实现。这涉及理解和实现调试适配器协议。在[此处](https://microsoft.github.io/debug-adapter-protocol/overview#How_it_works)可以找到更多细节。

## 调试器扩展package.json的解析

除了提供调试器独有的调试器适配器实现，调试器扩展需要`package.json`贡献各种调试有关的贡献点。

所以，让我们凑近看看演示调试的`package.json`。

如所有VS Code扩展，`package.json`声明了扩展的基本属性，**名称**、**发布者**和**版本**。使用**分类**字段使得扩展在VS Code扩展市场中更容易被发现。

```json
{
    "name": "mock-debug",
    "displayName": "Mock Debug",
    "version": "0.24.0",
    "publisher": "...",
    "description": "Starter extension for developing debug adapters for VS Code.",
    "author": {
        "name": "...",
        "email": "..."
    },
    "engines": {
        "vscode": "^1.17.0",
        "node": "^7.9.0"
    },
    "icon": "images/mock-debug.icon.png",
    "categories": ["Debuggers"],
    
    "contributes": {
        "breakpoints": [{ "language": "markdown" }],
        "debuggers": [
            {
                "type": "mock",
                "label": "Mock Debug",

                "program": "./out/mockDebug.js",
                "runtime": "node",

                "configurationAttributes": {
                    "launch": {
                        "required": ["program"],
                        "properties": {
                            "program":{
                                "type": "string",
                                "description": "Absolute path to a text file.",
                                "default": "${workspaceFolder}/${command:AskForProgramName}"
                            },
                            "stopEntry": {
                                "type": "boolean",
                                "description": "Automatically stop after launch.",
                                "default": true
                            }
                        }
                    }
                },
                
                "initialConfigurations": [
                    {
                        "type": "mock",
                        "request": "launch",
                        "name": "Ask for a file name",
                        "program": "${workspaceFolder/${command:AskForProgramName}}",
                        "stopOnEntry": true
                    }
                ],
                "configurationSnippets": [
                    {
                        "label": "Mock Debug: Launch",
                        "description": "A new configuration for launching a mock debug program",
                        "body": {
                            "type": "mock",
                            "request": "launch",
                            "name": "${2:Launch Program}",
                            "program": "^\"\\{workspaceFolder}/${1:Program}\""
                        }
                    }
                ],
                
                "variables": {
                    "AskForProgramName": "extension.mock-debug.getProgramName"
                }
            }
        ]
    },
    
    "activationEvents": ["onDebug", "onCommand:extension.mock-debug.getProgramName"]
}
```

现在看看包含调试扩展指定的贡献**contributes**字段。

受限，我们使用**breakpoints**贡献点列出了我们设置断点将会有效的语言。没有这个，就不可能为Markdown文件设置断点。

下面是**debuggers**字段。在此，通过调试**type**`mock`引入一种调试器。用户可以在启动配置中引用此类型。可选的属性**label**可以用于给调试类型一个好听的名字，让它在UI中显示。

因为调试扩展使用调试适配器，**program**属性中给出了其代码的相对路径。为了使扩展独立，应用程序必须处于扩展文件夹中。一般，我们将此应用程序放在叫做`out`或者`bin`的文件夹中，但是您可以随意使用不同的名字。

由于VS Code运行在不同的平台，我们必须确保DA程序也支持不同的平台。出于此，我们有下面的选项：

1. 如果程序是用平台独立的方式实现，比如作为运行在所有受支持平台可用的运行时上，您可以通过**runtime**属性指定运行时。对今天而言，VS Code支持`node`和`mono`运行时。我们上面的演示调试适配器使用此方式。

2. 如果您的DA实现需要不同平台上不同的可执行文件，可以用**program**属性指定平台：

```json
"debuggers": [{
    "type": "gdb",
    "windows": {
        "program": "./bin/gdbDebug.exe"
    },
    "osx": {
        "program": "./bin/gdbDebug.sh"
    },
    "linux": {
        "program": "./bin/gdbDebug.sh",
    }
}]
```

3. 也可以结合两种方法。下面的例子来自Mono DA，它作为mono应用程序实现，需要macOs和Linux的运行时，但不支持Windows：

```json
"debuggers": [{
    "type": "mono",
    "program": "./bin/monoDebug.exe",
    "osx": {
        "runtime": "mono"
    },
    "linux": {
        "runtime": "mono"
    }
}]
```

**configurationAttributes**声明了`launch.json`对调试器可用属性的框架。这个框架用于在编辑启动配置时校验`lauch.json`，支持智能感知和悬停帮助。

**initialConfigurations**定义了默认调试器`launch.json`的初始内容。这个信息在项目没有`launch.josn`时使用，当用户启动调试会话或在运行与调试视图中选择**创建launch.json**链接时使用。在这种情况下VS Code让用户选择调试环境，随后创建相应的`launch.json`：

![](https://code.visualstudio.com/assets/api/extension-guides/debugger-extension/debug-init-config.png)

相较于在`package.json`中静态定义`launch.json`初始内容，也可以通过实现`DebugConfigurationProvider`(详情查看[下面的使用DebugConfigurationProvider](#))动态计算初始配置。

**configurationSnippets**定义了启动配置片段，能够在编辑`launch.json`时出现在智能感知中。一般，使用调试环境名称作为片段`label`属性的前缀，以便在许多片段方案呈现时清晰辨认。

**variables**贡献了“变量”到“命令”的绑定。通过使用**${command:xyz}**词法，这些变量可以在启动配置中使用，当调试会话启动后，会被绑定的命令返回值取代。

命令的实现在扩展中，它可以是不具有UI的简单表达式，也可以是基于扩展API的UI特性的复杂功能。演示扩展绑定变量`AskForProgramName`到命令`extension.mock-debug.getProgramName`。此命令的[实现](https://github.com/microsoft/vscode-mock-debug/blob/606454ff3bd669867a38d9b2dc7b348d324a3f6b/src/extension.ts#L21-L26)位于`src/extension.ts`，使用`showInputBox`让用户输入程序名称：

```typescript
vscode.command.registerCommand('extension.mock-debug.getProgramName', config => {
    return vscode.window.showInputBox({
        placeHolder: 'Please enter the name of a markdown file in the workspace folder', 
        value: 'readme.md'
    });
});
```

变量可以作为**${command:AskForProgramName}**在启动配置中任何字符串输入值使用。

## 使用DebugConfigurationProvider

如果`package.json`基本的静态调试配置并不满足，就用`DebugConfigurationProvider`动态控制下述调试扩展的各个方面：

- 新创建launch.json的初始调试配置可以动态生成，比如基于工作空间可用的上下文信息。
- 启动配置可以在用于启动新调试会话前*处理*（或修改）。这允许基于工作空间可用信息填充默认值。存在两种*处理*方法：在启动配置变量被填充前调用的`resolveDebugConfiguration`，和在所有变量被替换后调用的`resolveDeubgConfigurationWithSubstitutedVariables`。如果校验逻辑插入额外值进入调试配置中，必须使用前者。如果校验逻辑需要访问所有调试配置属性的最终值，必须使用后者。

`src/extension.ts`中的`MockConfigurationProvider`实现了`resolveDebugConfiguration`检查调试扩展启动时没有launch.json存在，但是活跃编辑器中打开了Markdown文件的情况。这是一个典型的用户在编辑器中打开文件但只是想要调试而不需创建launch.json的场景。

供特定调试类型的调试配置提供器通过`vscode.debug.registerDebugConfigurationProvider`注册，通常在扩展的`activate`函数中。为了确保`DebugConfigurationprovider`已经在适当的时候注册，扩展必须在调试功能使用时及时启用。可以为`package.json`的`onDebug`事件配置扩展启用做到。

```json
"activationEvents": [
    "onDebug",
    // ...
]
```

这个总揽全局的`onDebug`会在使用一切调试功能时触发。只要扩展有较轻的启动负担，这就是不错的解决方案（即它不需要把大部分时间花在启动上）。如果调试扩展启动较重（例如因为要启动语言服务端），`onDebug`启用事件就会对其他调试扩展造成恶劣影响，这是因为它较早触发，并且没有考虑任何特定的调试类型。

对于重型调试扩展，更好的方法是使用良好分割的启用事件：

- `onDebugInitialConfigurations`恰好在`DebugConfigurationProvider`的`provideDebugConfigurations`方法之前触发。
- `onDebugResolve:type`恰好在`resolveDebugConfiguration`或者特定类型`DebugConfigurationProvider`的`resolveDeubgConfigurationWithSubstitutedVariables`之前触发。

**经验法则**：如果调试扩展启用不需要花费太多，使用`onDebug`。反之，基于`DebugConfigurationProvider`是否实现了相应的`provideDebugConfigurations`和/或`resolveDebugConfiguration`方法，使用`onDebugInitialConfigurations`和/或`onDebugResolve`。

## 发布您的调试器扩展
