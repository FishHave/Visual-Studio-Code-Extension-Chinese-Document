# 任务提供器

一般用户在一个叫`tasks.json`的文件中定义Visual Studio Code[任务](https://code.visualstudio.com/docs/debugtest/tasks)。但是，使用任务提供器，在软件开发的过程中，一些任务可以自动被VS Code扩展检测到。从VS Code运行**任务：运行任务**命令后，所有活动的任务提供器将贡献用户可以运行的任务。`tasks.json`文件让用户手动为指定的文件夹或工作空间定义任务，而任务提供器可以检查工作空间的详细信息，并自动创建关联的VS Code任务。例如，任务提供器可以检查是否有特定的构建文件，比如`make`或者`Rakefile`，并创建构建任务。本主题介绍了扩展如何自动检测并提供任务给终端用户。

本指南教学如何构建任务提供器自动检测[Rakefiles](https://ruby.github.io/rake/)的任务。完整源代码位于https://github.com/microsoft/vscode-extension-samples/tree/main/task-provider-sample。

## 任务定义
为了在系统中标识一个任务，贡献任务的扩展需要定义标识任务的属性。在Rake示例中，任务定义长这样：

```json
"taskDefinitions": [
    {
        "type": "rake",
        "required": [
            "task"
        ],
        "properties": {
            "task": {
                "type": "string",
                "description": "The Rake task to customize"
            },
            "file": {
                "type": "string",
                "description": "The Rake file that provides the task. Can be omitted."
            }
        }
    }
]
```

这贡献了`rake`任务的定义。任务定义具有`task`和`file`两个属性。`task`为Rake任务的名称，`file`指向包含任务的`Rakefile`。`task`属性是必需的，而`file`属性是可选的。如果`file`属性被忽略，会使用工作空间文件夹根目录下的`Rakefile`。

### 时句
任务定义可以拥有`when`属性。`when`属性指定了在何种条件下，此类型任务可用。`when`属性表现的和[VS Code的其他地方](/9.%20参考/6.%20时句上下文.md)的`when`属性一样。在创建任务定义时，应当考虑下面的上下文：

- `shellExcutionSupported`: 真当VS Code可以运行`ShellExecution`任务时，比如当VS Code作为桌面应用程序运行，或者使用一种远程扩展，比如Dev Containers。
- `processExecutionSupported`：真当VS Code可以运行`ProcessExecution`任务时，比如当VS Code作为桌面应用程序运行，或者使用一种远程扩展，比如Dev Containers。当前，它和`shellExcecutionSupported`永远具有相同的值。
- `customExecutionSupported`：真当VS Code可以运行`CustomExecution`时。这永远为真。

## 任务提供器
与提供代码补全的语言提供器类似，扩展可以注册任务提供器推演所有可用的任务。可以如下面的代码片段使用`vscode.tasks`命名空间完成：
```typescript
import * as vscode from 'vscode';

let rakePromise: Thenable<vscode.Task[]> | undefined = undefined;

const taskProvider = vscode.tasks.registerTaskProvider('rake', {
    provideTasks: () => {
        if (!rakePromise) {
            rakePromise = getRakeTasks();
        }
        return rakePromise;
    },
    resolveTask(_task: vscode.Task): vscode.Task | undefined {
        const task = _task.definition.task;
        // 如RakeTaskDefinition所指定的一样，Rake任务包括任务和可选的文件。
        // 通过检查是否存在任务，以确保这看起来像Rake任务
        if (task) {
            // resolveTask 共用使用的定义。
            const definition: RakeTaskDefinition = <any>_task.definition;
            return new vscode.Task(
                definition,
                _task.scope ?? vscode.TaskScope.Workspace,
                definition.task,
                'rake',
                new vscode.ShellExecution(`rake ${definition.task}`)
            );
        }
        return undefined;
    }
});
```

如`provideTasks`，`resolveTask`方法由VS Code调用以获取来自扩展的任务。`resolveTask`可以代替`provideTasks`被调用，目的是为实现它的提供器带来更优秀的性能选项。例如，如果用户有运行提供任务的扩展的按键绑定，由VS Code调用任务提供器的`resolveTask`，快速获得任务会比调用`provideTasks`并等待扩展提供所有的任务更好。创建设置让用户关闭独立任务提供器是一个好的做法，所以也很普遍。用户可能会注意到来自特定提供器的任务获取得比较慢，然后就关闭提供器。在这种情况下，用户也许仍然在`task.json`引用了来自此提供器的任务。如果没有实现`resolveTask`，会有警告提示`tasks.json`中的任务未创建。具有`resolveTask`的扩展仍然可以提供定义在`tasks.json`中的任务。

`getRakeTasks`实现做了下面的事情：

- 对每个工作空间文件夹使用`rake -AT -f Rakefile`命令，列出所有定义在`Rakefile`Rake任务。
- 解析标准输出流。
- 对每个列出的任务，创建`vscode.Task`实现。

由于Rake测试初始化需要在`package.json`文件中定义任务，所有VS Code使用TypeScript接口定义了此结构，如下：
```typescript
interface RakeTaskDefinition extends vscode.TaskDefinition {
    /**
     * 任务名
     */
    task: string;

    /**
     * 包含任务的Rake文件
     */
    file?: string;
}
```

假设在第一个工作空间中，来自一个任务的输出为`compile`，关联的任务创建过程如下：

```typescript
let task = new vscode.Task(
    { type: 'rake', task: 'compile' },
    vscode.workspace.workspaceFolders[0],
    'compile',
    'rake',
    new vscode.ShellExecution('rake compile')
);
```

对输出中列出的每个任务，将使用上面的方式创建关联的VS Code任务，并返回来自`getRakeTasks`调用的所有任务的数组。

`ShellExecution`在OS特定的命令解释器中执行`rake compile`（例如在Windows下命令会在PowerShell执行，在Ubuntu中会在bash中执行）。如果任务应该直接执行某个进程（而不生成命令行解释器），就用`vscode.ProcessExcution`。`ProcessExecution`有一个优势，扩展可以完全控制传递给进程的参数。使用`ShellExecution`可以充分利用命令行解释（比如bash下的扩展通配符）。如果`ShellExecution`与一个命令行一同创建，扩展需要保证命令中引号和转义的正确使用（比如处理空格）。

## CustomExecution
通常，最好使用`ShellExecution`或者`ProcessExection`，因为它们很简单。但是如果您的任务在每次运行中保存很多状态，在单独的脚本或进程无法正常运作，或者需要对输出的额外处理，`CustomExecution`可能适合这种情况。`CustomExecution`的使用通常面向复杂的构建系统。`CustomExecution`仅具有在任务运行时执行的回调。这使它在扩展的功能方面具有更大的灵活性，但也意味着任务提供器需要对所有发生的进程管理和输出解析负责。任务提供器也对实现`Pseudoterminal`和将其从`CustomExecution`回调负责。

```typescript
return new vscode.Task(
    definition,
    vscode.TaskScope.Workspace,
    `${flavour} ${flags.join(' ')}`,
    CustomBuildTaskProvider.CustomBuildScriptType,
    new vscode.CustomExecution(
        async (): Promise<vscode.Psedoterminal> => {
            // 当任务执行时，将运行本回调。我们在此初始化任务运行。
            return new CustomBuildTaskTerminal(
                this.workspaceRoot,
                flavor,
                flags,
                () => this.sharedState,
                (state: string) => (this.sharedState = state)
            );
        }
    )
);
```

如需获取包括`Pseudoterminal`实现的完整示例，前往https://github.com/microsoft/vscode-extension-samples/tree/main/task-provider-sample/src/customTaskProvider.ts。
