# Web扩展
Visual Studio Code编辑器可以在浏览器中运行。比如说在GitHub浏览仓库，或者拉取请求，通过按下`.`（点键）进入`github.dev`用户界面。在Web使用VS Code，安装的扩展会在浏览器中的扩展宿主运行，叫做“Web扩展宿主”。可以在Web扩展宿主中运行的扩展叫做“Web扩展”。

Web扩展与常规扩展使用相同的结构，但考虑到不同的运行时，不会使用为Node.js运行时而编写的扩展的代码。Web扩展仍然可以访问完整的VS Code API，但不再能使用Node.js API和模块加载，而是被限制在浏览器沙盒中，因此，对比普通扩展有所[限制](#Web扩展主文件)。

Web扩展运行时也受VS Code桌面端支持。如果您决定创建一个Web扩展，它支持[Web VS Code](https://code.visualstudio.com/docs/setup/vscode-web)（包括`vscode.dev`和`github.dev`），也支持桌面端，和[GitHub Codespaces](https://code.visualstudio.com/docs/remote/codespaces)等服务。

## Web扩展详解
Web扩展的[结构类似常规扩展](/2.%20开始/2.%20扩展解剖.md)。扩展清单（`package.json`）定义扩展源代码的入口文件，以及声明扩展贡献。

对于Web扩展来说，[主入口文件](#Web扩展主文件)由`browser`属性定义，而不是与常规扩展一样由`main`属性定义。

`contributes`属性与常规扩展工作方式相同。

下面的例子展示了一个简单的Hello World扩展的`package.json`，这个扩展仅运行在Web扩展宿主（仅拥有`browser`入口）：
```json
{
    "name": "helloworld-web-sample",
    "displayName": "helloworld-web-sample",
    "description": "HelloWorld example for VS Code in the browser",
    "version": "0.0.1",
    "publisher": "vscode-samples",
    "repository": "https://github.com/microsoft/vscode-extension-samples/helloworld-web-sample",
    "engines": {
        "vscode": "^1.74.0"
    },
    "categories": ["Other"],
    "activationEvents": [],
    "browser": "./dist/web/extension.js",
    "contributes": {
        "commands": [
            {
                "command": "helloworld-web-sample.helloworld",
                "title": "Hello World"
            }
        ]
    },
    "scripts": {
        "vscode:prepublish": "npm run package-web",
        "compile-web": "webpack",
        "watch-web": "webpack --watch",
        "package-web": "webpack --mode production --devtool hidden-source-map"
    },
    "devDependencies": {
        "@types/vscode": "^1.59.0",
        "ts-loader": "^9.2.2",
        "webpack": "^5.38.1",
        "webpack-cli": "^4.7.0",
        "@types/webpack-env": "^1.16.0",
        "process": "^0.11.10"
    }
}
```

> **注意**：如果您的扩展面向高于1.74的VS Code，您必须在`activationEvents`中单独列出`onCommand:helloworld-web-sample.helloworld`

仅有`main`入口，而没有`browser`的扩展不是Web扩展。它们会被Web扩展宿主忽略，并且在扩展视图中不可下载。

![](https://code.visualstudio.com/assets/api/extension-guides/web-extensions/extensions-view-item-disabled.png)

仅有声明性贡献（仅`contributes`，没有`main`或者`browser`）可以是Web扩展。它们可以在[Web VS Code](https://code.visualstudio.com/docs/setup/vscode-web)中安装运行，并且不需要扩展作者作出任何修改。扩展声明性贡献的例子有主题、语法和代码片段。

扩展可以同时具有`browser`和`main`入口使其可以在浏览器和Node.js运行时运行。[更新已有的扩展为Web扩展](#更新已有的扩展为Web扩展)小节展示了如何将扩展迁移为能在两种运行时工作。

[Web扩展的启用](#Web扩展的启用)小节列举了用来决定扩展额能否在Web扩展宿主加载的条件。

### Web扩展主文件
Web扩展主文件由`browser`属性定义。Web扩展的脚本寄宿在[浏览器WebWorker](https://developer.mozilla.org/zh-CN/docs/Web/API/Web_Workers_API)环境。它被限制在浏览器Worker沙盒，相比运行在Node.js运行时的普通扩展有所限制。

- 不支持导入或需求其他模块。`importScripts`同样不可用。所以，代码必须打包为单个文件。
- [VS Code API](/9.%20参考/1.%20VS%20Code%20API.md)可以通过`require('vscode')`的方式加载。这仍然可以使用，因为我们给`require`开了后门，但是这个后门不可以用于加载额外的扩展文件或者Node模块。它只能使用`require('vscode')`。
- Node.js全局成员和库，比如`process`、`os`、`setImmediate`、`path`、`util`、`url`在运行时中不可用。然而，它们可以使用类似Webpack这样的工具添加。[Webpack配置](#Webpack配置)小节解释了这是如何做到的。
- 开放的工作空间或者文件夹在虚拟文件系统上。对工作空间文件的访问权限需要经过VS Code在`vscode-workspace-fs`[文件系统](/9.%20参考/1.%20VS%20Code%20API.md)API。
- [扩展上下文](https://code.visualstudio.com/api/references/vscode-api)位置（`ExtensionContext.extensionUri`）和存储位置（`ExtensionContext.storageUri`,`globalStorageUri`）也可以运作在虚拟文件系统，并且需要经过`vscode.workspace.fs`。
- 要访问Web资源，必须使用[Fetch](https://developer.mozilla.org/zh-CN/docs/Web/API/Fetch_API)。访问的资源需要支持[跨源资源共享](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Guides/CORS)（CORS）
- 创建子进程或者运行可执行文件是不可能的。然而，可以通过[Worker](https://developer.mozilla.org/en-US/docs/Web/API/Worker)API创建Web Worker。这用于如[Web扩展语言服务端协议](#Web扩展语言服务端协议)小节介绍的运行语言服务端。
- 作为常规扩展，`activate/deactivate`函数需要通过`exports.activate = ...`的方式导入。

## 开发Web扩展
谢天谢地，TypeScript和Webpack这样的工具可以避免许多浏览器运行时限制，允许您用常规扩展相同的方式编写Web扩展。Web扩展和常规可以可以生成自同一份源代码。


例如，由`yo code`[生成器](https://www.npmjs.com/package/generator-code)创建的`Hello Web Extension`仅在构建脚本方面有所不同。通过使用**调试：选择并开始调试**命令提供的提供的启动配置，您可以像传统Node.js扩展一样，运行和调试生成的扩展。

## 创建Web扩展
为了搭建新的Web扩展，使用`yo code`，选取**New Web Extension**。确保`generator-code`是最新版本（>=generator-code@1.6）。如需更新生成器和yo，运行`npm i -g yo generator-code`。

创建扩展由扩展源代码（显示hello world通知的命令）、`package.json`清单文件和webpack或者esbuild配置文件组成。

让事情简单一些，我们假定您使用`webpack`作为打包工具。在文章的最后，我们会解释选择`esbuild`会有什么不同。

- `src/web/extension.ts`是扩展的入口源代码文件。这和常规的hello扩展是显著的不同之处。
- `package.json`是扩展清单
    - 它使用`browser`属性指向入口文件
    - 它提供脚本：`compile-web`、`watch-web`和`package-web`，用来编译、观测和打包。
- `webpack.config.js`是webpack配置文件，编译并打包扩展源代码为单个文件。
- `.vscode/launch.json`包含运行Web扩展的启动配置和VS Code桌面端Web扩展宿主的测试（不再需要设置`extensions.webWorker`）。
- `.vscode/task.json`包括启动配置使用的构建任务。它使用`npm run watch-web`，并依赖于webpack特别的`ts-webpack-watch`问题配对器。
- `.vscode/extensions.json`包括提供问题配对其的扩展。这些扩展需要安装使启动配置能够工作。
- `tsconfig.json`定义匹配`webworker`运行时的编译选项。

[helloworld-web-sample](https://github.com/microsoft/vscode-extension-samples/tree/main/helloworld-web-sample)源代码与生成器创建的那个相同。

### Webpack配置
Webpack配置文件由`yo code`自动生成。它从您的扩展中打包源代码为单个JavaScript文件，并在Web扩展宿主中加载。

之后我们会解释如何使用esbuild作为打包工具，但是我们从Webpack开始

[webpack.config.js](https://github.com/microsoft/vscode-extension-samples/blob/main/helloworld-web-sample/webpack.config.js)

```javascript
const path = require('path');
const webpack = require('webpack');

/** @typedef {import('webpack').Configuration} WebpackConfig **/
/** @type WebpackConfig */
const webExtensionConfig = {
    mode: 'none', // 这让源代码尽量与原版接近（打包时我们将其设置为'production'）
    target: 'webworker', // 扩展运行在Webworker上下文
    entry: {
        extension: './src/web/extension.ts', // Web扩展主文件的源
        'test/suite/index': './src/web/test/suite/index.ts' // Web扩展测试运行器的源
    },
    output: {
        filename: '[name].js',
        path: path.join(__dirname, './dist/web'),
        libraryTarget: 'commonjs',
        devtoolModuleFilenameTemplate: '../../[resource-path]'
    },
    resolve: {
        mainFields: ['browser', 'module', 'main'], // 在导入的Node模块中查找`browser`入口
        extensions: ['.ts', '.js'], // 支持ts文件和js文件
        alias: {
            // 提供Node模块和源文件的其他实现
        },
        fallback: {
            // Webpack 5 不再自动替代Node.js的核心模块
            // 参见 https://webpack.js.org/configuration/resolve/#resolvefallback
            // 以获取Node.js核心模块的替代列表
            assert: require.resolve('assert')
        }
    },
    module: {
        rules: [
            {
                test: /\.ts$/,
                exclude: /node_modules/,
                use: [
                    {
                        loader: 'ts-loader'
                    }
                ]
            }
        ]
    },
    plugins: [
        new webpack.ProvidePlugin({
            process: 'process/browser' // 提供全局`process`变量的后门
        })
    ],
    externals: {
        vscode: 'commonjs vscode' // 忽略，因为它不存在
    },
    performance: {
        hints: false
    },
    devtool: 'nosources-source-map' // 创建源映射，指向原版源代码
};
module.exports = [webExtensionConfig];
```

`webpack.config.js`一些重要字段包括：
- `entry`字段包括扩展和测试集主入口文件
    - 您也许需要调整路径，使之正确指向您正在用于`package.json`的`main`的文件。
    - 如果您不想要打包测试，您可以略去测试集字段。
- `output`字段指明编译的文件应当位于哪里。
    - `[name]`会被`entry`中的键替换。所以在生成的配置文件中，它会产生`dist/web/extension.ts`和`dist/web/test/suite/index.js`。
- `target`字段告知了编译后的JavaScript文件会运行在哪种环境。对于Web扩展，您设置此为`webworker`。
- `resolve`字段包括添加不受浏览器支持的Node库的变种与回退的功能。
    - 如果您正在使用`path`之类的库，您可以指定如何在Web已编译上下文处理`path`。例如，您可以指向项目中一个用`path: path.resolve(__dirname, 'src/my-path/implementation-for-web.js')`定义`path`的文件。或者，您可以使用Browserify Node打包版本的库，叫做`path-browserify`，并指定`path: require.resolve('pathb-browserify')`。
    - 参加[webpack resolve.fallback](https://webpack.js.org/configuration/resolve/#resolvefallback)获得Node.js核心模块替代列表。
- `plugin`部分使用[DefinePlugin插件](https://webpack.js.org/plugins/define-plugin/)替换全局成员，比如`process`Node.js全局成员。

## 测试您的Web扩展
在发布到市场前，有三种方法测试Web扩展。
- 使用VS Code桌面端，并启用`--extensionDevelopmentKind=web`选项，在VS Code中运行Web扩展宿主使用Web扩展。
- 使用`@vscode/test-web`Node模块打开浏览器容纳装载扩展的Web VS Code，以此作为本地服务端。
- 在[vscode.dev](https://vscode.dev/?vscode-lang=zh-cn)[侧面加载](#在vscode.dev测试您的Web扩展)，看看在实际环境中您的扩展。

### 在VS Code桌面端测试您的扩展
为了保留当下的VS Code扩展开发体验，VS Code桌面端支持在常规的Node.js扩展宿主之外，运行Web扩展宿主。

使用由**New Web Extension**生成器提供的`pwa-extensionhost`启动配置：

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Run Web Extension in VS Code",
            "type": "pwa-extensionHost",
            "debugWebWorkerHost": true,
            "request": "launch",
            "args": [
                "--extensionDevelopmentPath=${workspaceFolder}$",
                "--extensionDevelopmentKind=web"
            ],
            "outFiles": ["${workspaceFolder}/dist/web/**/*.js"],
            "preLaunchTask": "npm: watch-web"
        }
    ]
}
```

它使用`npm: watch-web`任务，调用`npm run watch-web`编译扩展。那个任务期望在`task.json`中：
```json
{
    "version": "2.0.0",
    "tasks": [
        {
            "type": "npm",
            "script": "watch-web",
            "group": "build",
            "isBackground": true,
            "problemMatcher": ["$ts-webpack-watch"]
        }
    ]
}
```

`$ts-webpack-watchs`是问题配对器，它可以解析来自webpack的输出。它由[TypeScript + Webpack问题配对器](https://marketplace.visualstudio.com/items?itemName=amodio.tsl-problem-matcher)扩展提供。

在启动的**扩展开发宿主**实例中，Web扩展会启用，并在Web扩展宿主上运作。运行`Hello World`命令启用扩展。

打开**运行中的扩展**视图（命令：**开发者：显示运行中的扩展**）查看正在运行在Web扩展宿主中的扩展。

### 使用@vscode/test-web在浏览器中测试您的扩展

[@vscode/test-web]Node模块提供CLI和API在浏览器中测试扩展。
Node模块贡献了一个npm二进制库`vscode-test-web`，它可以在命令行中打开Web VS Code：
- 它会下载VS Code的Web信息，进入`.vscode-test-web`。
- 在`localhost:3000`上启动本地服务端。
- 打开浏览器（Chromium、FireFox或者Webkit）。

您可以从命令行中运行之：
```bash
npx @vscode/test-web --extensionDevelopmentPath=$extensionFolderPath $testDataPath
```
或者这样更好：添加`@vscode/test-web`作为您的扩展的开发依赖，将其作为脚本调用：

```json
{
    "devDependencies": {
        "@vscode/test-web": "*"
    },
    "scripts": {
        "open-in-browser": "vscode-test-web --extensionDevelopmentPath=. ."
    }
}
```

检查[@vscode/test-web README](https://www.npmjs.com/package/@vscode/test-web)获取更多CLI选项信息

| 选项 | 参数介绍 |
| - | - |
|--browserType | 启动的浏览器：`chromium`（默认），`firefox`或者`webkit` |
|--extensionDevelopmentPath | 指向正在开发中的扩展的路径 |
|--extensionTestPath | 指向要运行的测试模块的路径 |
|--permission | 打开的浏览器获得的权限：比如`clipboard-read`、`clipboard-write`。参加[选项的完整列表](https://playwright.dev/docs/api/class-browsercontext#browser-context-grant-permissions)。参数可以多次提供。 |
|--folder-uri | 打开VS Code所处的工作空间URI。提供`folderPath`时忽略 |
|--extensionPath | 指向包含额外扩展的文件夹。参数可以多次提供 |
|folderPath | 打开VS Code所处的本地文件夹。文件夹的内容会作为虚拟文件系统使用，并且作为工作空间打开。 |

VS Code的Web信息会下载到`.vscode-test-web`文件夹。您要添加到您的`.gitignore`文件。

### 在vscode.dev中测试您的Web扩展
在您发布您的扩展供所有VS Code Web用户使用前，您可以验证您的扩展在实际的`[vscode-dev](https://vscode.dev/)环境下的表现。

要看到扩展在vscode.dev上的表现，您首先要让其寄宿在您的及其上，以便vscode.dev能够下载并运行之。

首先，您需要[install `mkcert`](https://github.com/FiloSottile/mkcert#installation)。

随后，在您不会找不到的位置生成`localhost.pem`和`localhost-key.pem`文件（例如`$HOME/certs`）：
```text
$ mkdir -p $HOME/certs
$ cd $HOME/certs
$ mkcert -install
$ mkcert localhost
```

然而，在您的扩展路径，运行`npx serve`启动一个HTTP服务端：

```text
$ npx serve --cors -l 5000 --ssl-cert $HOME/certs/localhost.pem --ssl-key $HOME/certs/localhost-key.pem
npx: installed 78 in 2.196s

   ┌────────────────────────────────────────────────────┐
   │                                                    │
   │   Serving!                                         │
   │                                                    │
   │   - Local:            https://localhost:5000       │
   │   - On Your Network:  https://172.19.255.26:5000   │
   │                                                    │
   │   Copied local address to clipboard!               │
   │                                                    │
   └────────────────────────────────────────────────────┘
```

最后，打开[vscode.dev](https://vscode.dev)，在命令面板(<kbd>Ctrl+Shift+P</kbd>)选择运行**开发者模式：从位置安装扩展...**，粘贴上面的URL，在本示例中为`https://localhost:5000`，选择**安装**。

## Web扩展测试
Web扩展支持测试，并且实现类似于常规扩展测试。参见[测试扩展](/7.%20测试与发布/1.%20测试扩展.md)文章了解扩展测试的基本结构。

`@vscode/test-web`Node模块等价于`@vscode/test-electron`（之前叫做`vscode-test`）。它允许您从Chromium、Firefox和Safari的命令行运行扩展测试。

它的功能分为以下几个步骤：

1. 从本地Web服务端启动Web VS Code编辑器。
2. 打开指定的浏览器。
3. 运行提供的测试运行器脚本。

您可以在持续开发中运行测试，以确保扩展在所有浏览器中工作。

运行在Web扩展宿主的测试运行器脚本与[Web扩展主文件](#Web扩展主文件)有相同的限制：
1. 所有的文件都打包为单个文件。它们应该包含测试运行器（例如：Mocha）和松油测试（通常为`*.test.ts`）。
- 仅支持`require('vscode')`。

`yo code`Web扩展生成器创建的[webpack配置](https://github.com/microsoft/vscode-extension-samples/blob/main/helloworld-web-sample/webpack.config.js)有一部分关于测试。它期望测试运行器脚本位于`./src/web/test/suite/index.ts`。它提供的[测试运行器脚本](https://github.com/microsoft/vscode-extension-samples/blob/main/helloworld-web-sample/src/web/test/suite/index.ts)使用Web版本的Mocha，包含webpack特殊语法导入所有测试文件。

```typescript
require('mocha/mocha'); // 导入mocha Web构建

export function run(): Promise<void> {
    return new Promise((c, e) => {
        mocha.setup({
            ui: 'tdd',
            reporter: undefined
        });

        // 打包所有当前文件夹匹配`*.test`的文件
        const importAll = (r: __WebpackModuleApi.RequireContext) => r.keys().forEach(r);
        importAll(require.context('.', true, /\.test$/));

        try {
            // 运行Mocha测试
            mocha.run(failures => {
                if (failures > 0) {
                    e(new Error(`${failures} tests failed`));
                } else {
                    c();
                }
            });
        } catch (err) {
            console.error(err);
            e(err);
        }
    });
}
```

添加下面的片段到`package.json`，并用`npm test`运行之，以从命令行运行Web测试。

```json
"devDependencies": {
    "@vscode/test-web": "*"
},
"scripts": {
    "test": "vscode-test-web --extensionDevelopmentPath=. --extensionTestssPath=dist/web/test/suite/index.js"
}
```
传递本地文件夹路径（`folderPath`）作为最后一个参数以让VS Code携带测试数据打开文件夹。

要在VS Code（内部版）桌面端运行（和调试）扩展测试，使用`Extension Tests in VS Code`启动配置：

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Extension Tests in VS Code",
            "type": "extensionHost",
            "debugWebWorkerHost": true,
            "request": "launch",
            "args": [
                "--extensionDevelopmentPath=${workspaceFolder}",
                "--extensionDevelopmentKind=web",
                "--extensionTestsPath=${workspaceFolder}/dist/web/test/suite/index"
            ],
            "outFiles": ["${workspaceFolder}/dist/web/**/*.js"],
            "preLaunchTask": "npm: watch-web"
        }
    ]
}
```

## 发布Web扩展
Web扩展与其他扩展一样，都寄存在[市场](https://marketplace.visualstudio.com/vscode)。

确保使用最新版本的`vsce`发布您的扩展。`vsce`会给所有Web扩展打标签。为此，`vsce`遵循在[Web扩展的启用](#Web扩展的启用)小节列举的规则。

## 更新已有的扩展为Web扩展
### 不具代码的扩展
不具代码，仅有贡献点（例如主题、代码片段和基本语言扩展）的扩展不需要任何修改。它们可以在Web扩展宿主运行，并且可以从扩展视图安装。

不需要重新发布，但是发布新版本的扩展时，确保使用最新版本的`vsce`。

### 迁移具有代码的扩展

具有源代码（由`main`属性定义）的扩展需要提供[Web扩展主文件](#web扩展主文件)，并在`package.json`中设置`browser`属性。

遵从下面步骤重新编译您的扩展代码，适配浏览器环境：

- 如[Webpack配置](#webpack配置)小节展示的那样，添加Webpack配置文件。如果您已经有供您的Node.js扩展代码使用的Webpack文件，您可以添加一个新的以适应Web环境。查阅[vscode-css-formatter](https://github.com/aeschli/vscode-css-formatter/blob/master/webpack.config.js)作为典例。
- 同[测试您的Web扩展](#测试您的web扩展)小节所展示，添加`launch.json`和`tasks.json`文件。
- 在Webpack配置文件，设置输入文件为现有的Node.js主文件或者创建新的Web扩展主文件。
- 同[Web扩展详解](#web扩展详解)小节所说，在`package.json`中，添加`browser`和`scripts`属性。
- 运行`npm run compile-web`调用webpack，并查看还有哪里需要处理以使您的扩展在Web工作。

要确保金额可能多的源代码能够复用，这里有几个技巧：

- 添加项目到[resolve.fallback](https://webpack.js.org/configuration/resolve/#resolvefallback)中，以替代Node.js核心模块，比如`path`。
- 使用[DefinePlugin插件](https://webpack.js.org/plugins/define-plugin/)以提供Node.js全局成员，如`process`。
- 使用同时工作在浏览器和Node运行时的Node模块。Node模块可以通过定义`browser`和`main`入口实现该目标。Webpack会自动使用其目标配对。作为例子的Node模块有[request-light](https://github.com/microsoft/node-request-light)和[@vscode/l10n](https://github.com/microsoft/vscode-l10n)。
- 使用[resolve-alias](https://webpack.js.org/configuration/resolve/#resolvealias)，提供Node模块或源代码的替代实现。
- 分割代码为浏览器部分和Node.js普通和通用部分。通用部分仅使用同时在浏览器和Node.js都运行的代码。创建功能的抽象，让其在Node.js和浏览器中具有不同的实现。
- 查找`path`、`URI.file`、`context.extensionPath`、`rootPath.uri.fsPath`的用法。因为它们仅适用Web VS Code，而在虚拟工作空间（非文件系统）不起作用。使用`URI.parse`、`context.extensionUri`中的URI替代。[vscode-uri](https://www.npmjs.com/package/vscode-uri)Node模块提供`joinPath`、`dirName`、`baseName`、`extName`、`resolvePath`。
- 查找`fs`的用法。使用VS Code的`workspace.fs`替代。

当扩展在Web中运行时，提供较少功能是没关系的。使用[时句上下文](/9.%20参考/6.%20时句上下文.md)控制命令、视图和任务在Web虚拟工作空间中是启用还是隐藏。
- 使用`virtualWorkspace`上下文变量探明当前工作空间是否是非文件系统工作空间。
- 使用`resourceScheme`检查当前资源是否是`file`资源。
- 如果有存在的平台Shell，使用`shellExecutionSupported`。
- 实现额外的命令处理器，显示对话框解释为什么命令不可用。

WebWorker可以用于分支处理的替代品。我们已经更新了几个语言服务端，使之能在Web扩展中运行，包括内建的[JSON](https://github.com/microsoft/vscode/tree/main/extensions/json-language-features)
、[CSS](https://github.com/microsoft/vscode/tree/main/extensions/css-language-features)和[HTML](https://github.com/microsoft/vscode/tree/main/extensions/html-language-features)语言服务端。下面的[语言服务端协议](#语言服务端协议)小节提供了更多细节。

浏览器运行时环境仅支持运行JavaScript和[WebAssembly](https://webassembly.org/)。使用其他变成语言编写的库需要跨编译，例如使用工具将[C/C++](https://developer.mozilla.org/en-US/docs/WebAssembly/Guides/C_to_Wasm)和[Rust](https://developer.mozilla.org/en-US/docs/WebAssembly/Guides/Rust_to_Wasm)编译为WebAssembly。
### Web扩展语言服务端协议

[vscode-languageserver-node]是[语言服务端协议]（LSP）的实现，它用于语言服务端的基础实现，如[JSON](https://github.com/microsoft/vscode/tree/main/extensions/json-language-features)
、[CSS](https://github.com/microsoft/vscode/tree/main/extensions/css-language-features)和[HTML](https://github.com/microsoft/vscode/tree/main/extensions/html-language-features)。

从3.16.0开始，客户端和服务端也提供浏览器实现。服务端可以在Web Worker中运行，并且连接基于Web Worker的`postMessage`协议。

浏览器的客户端可以在'vscode-languageclient/browser'找到：
```typescript
import { LanguageClient } from `vscode-languageclient/browser`;
```
服务端位于`vscode-languageserver/browser`。

[lsp-web-extension-sample](https://github.com/microsoft/vscode-extension-samples/tree/main/lsp-web-extension-sample)展示了其工作方式
