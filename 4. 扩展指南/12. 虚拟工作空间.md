# 虚拟工作空间
一些扩展，比如说[GitHub Repositories](https://marketplace.visualstudio.com/items?itemName=GitHub.remotehub)扩展，在由一个[文件系统提供者](/4.%20扩展指南/11.%20虚拟文档.md)支撑的一个或多个文件夹打开VS Code。当扩展实现了文件系统提供者，工作空间资源也许不会处于本地磁盘，但是是**虚拟**的，处在服务器或云端中，编辑操作也发生在此。

这项配置叫做**虚拟工作空间**。VS Code窗口打开虚拟空间，左下角的远程提示标签就会显示，类似于其他[远程开发](https://code.visualstudio.com/docs/remote/remote-overview)。

![](https://code.visualstudio.com/assets/api/extension-guides/virtual-workspaces/remote-indicator.png)

并非所有的扩展都能处理虚拟资源，它们需要资源位于磁盘上。一些扩展使用依赖于磁盘访问权限的工具、需要同步文件访问或者没必要使用抽象文件系统。在这些情况，VS Code打开虚拟工作空间时，会提示用户它们正在受限模式下运行，一些扩展被禁用或者功能具有限制。

通常来说，在浏览和编辑远程资源时，用户希望尽可能多的扩展在虚拟空间工作来获取更好的用户体验。本篇指南展示了扩展如何在虚拟工作空间下测试，介绍一些允许它们在虚拟工作空间下工作的修改，并引入`virtualWorkspaces`功能属性。

修改扩展使之适配虚拟工作空间同样是与[Web VS Code](https://code.visualstudio.com/docs/setup/vscode-web)打交道的重要一步。Web VS Code仅在浏览器下运行，由于浏览器沙盒的原因，其工作空间是虚拟的。参见[Web扩展](/4.%20扩展指南/13.%20Web扩展)指南获取更多信息。

## 我的扩展会受影响吗？

如果扩展没有可执行代码，只有声明，如主题、按键绑定、代码片段或者语法扩展，它可以在虚拟工作空间上运行，并且不需要任何修改。

有代码的扩展，也就是定义了`main`入口的扩展，需要检查和可能的修改。

## 在虚拟工作空间运行扩展

安装[GitHub Repositories]扩展，从命令面板运行**打开GitHub仓库...**命令。该命令会显示快速选取下拉菜单，您可以粘贴任意GitHub URL，或选择寻找特定的仓库或拉取请求。

这将打开一个在虚拟工作空间上的VS Code窗口，在此，所有资源都是虚拟的。

## 检查扩展代码是否对虚拟资源准备

VS Code API支持虚拟文件系统已经有一会了。您可以查看[文件系统提供者API](/4.%20扩展指南/11.%20虚拟文档.md)。

文件系统提供者为URI框架而注册（例如，`vscode-vfs`），该文件系统上的资源会由使用此框架的URI代表。

检查您的扩展是否处理了VS Code API返回的URI：

1. 不要假定URI框架是`file.URI.fsPath`，这个框架仅在URI框架为`file`时可使用。

2. 留意`fs`node模块对文件系统操作的的用法。如果可能，使用`vscode.workspace.fs`API，此API将操作委托给合适的文件系统提供者。

3. 检查依赖`fs`访问权限的第三方组件（例如，语法服务器或node模块）。

4. 如果您从命令运行了可执行文件或者任务，检查这些命令在虚拟工作空间窗口是否有效，或者它们是否应该被禁用。

## 标明您的扩展是否能够处理虚拟工作空间

`package.json`中`capabilities`下的`virtualWorkspaces`属性用于标识您的扩展是否能处理虚拟工作空间。

### 不支持虚拟工作空间

下面的例子声明了一个不支持虚拟工作空间，且不应在启动时启用的扩展。

```json
{
    "capabilities": {
        "virtualWorkspaces": {
            "supported": false,
            "description": "调试在虚拟工作空间不可用。"
        }
    }
}
```

### 部分和全部支持工作空间

如果扩展支持或部分支持虚拟工作空间，应当定义`"virtualWorkspaces": true`。

```json
{
    "capabilities": {
        "virtualWorkspaces": true
    }
}
```

如果扩展能够运作，但功能有限制，应当解释限制给用户：

```json
{
    "capabilities": {
        "virtualWorkspaces": {
            "supported": "limited",
            "description": "在虚拟工作空间中，在文件间分析和查找引用不受支持。"
        }
    }
}
```

扩展视图会显示介绍：

![](https://code.visualstudio.com/assets/api/extension-guides/virtual-workspaces/extensions-view.png)

扩展应当随后如介绍的禁用不在虚拟工作空间支持的功能。

### 默认
`"virtualWorkspaces": true`是所有没有填写该功能属性的扩展的默认值。

然而，在测试虚拟空间是，我们想到了一系列我们认为应当在虚拟工作空间禁用的扩展。该列表可以在[issue #122836](https://github.com/microsoft/vscode/issues/122836)找到。这些扩展以`"virtualWorkspaces": false`为默认值。

当然，扩展作者是最好做决定的。扩展的`package.json`中`virtualWorkspaces`功能会覆盖我们的默认值，我们最终会停止使用我们的列表。

## 当虚拟工作空间打开时禁用功能
### 禁用命令和视图贡献

命令和视图及其他贡献的可用与否由[时句](/9.%20参考/6.%20时句上下文.md)上下文键控制。

`virtualWorkspace`上下文键设置为所有工作空间文件夹是否位于虚拟文件系统。下面的例子仅展示了不在虚拟工作空间的命令面板中的命令`npm.publish`：

```json
{
    "menus": {
        "commandPalette": [
            {
                "command": "npm.publish",
                "when": "!virtualWorkspace"
            }
        ]
    }
}
```

`resourceScheme`上下文键设置为文件浏览器中当前元素或编辑器打开的元素的URI。

在下面的例子中，`npm.runSelectedScript`命令仅在依赖的资源位于本地磁盘时显示在编辑器上下文菜单。

```json
{
    "menus": {
        "editor/context": [
            {
                "command": "npm.runSelectedScript",
                "when": "resourceFilename == 'package.json' && resourceScheme == file"
            }
        ]
    }
}
```

## 程序检测虚拟工作空间

要检查当前的工作空间是否由非`file`框架组成，或是虚拟的，您可以使用下面的源代码：

```typescript
const isVirtualWorkspace =
    workspace.workspaceFolders &&
    workspace.workspaceFolders.every(f => f.uri.scheme !== 'file');
```

## 语言扩展与虚拟工作空间
### 语言空间支持虚拟工作需要达到什么

所有扩展都能完全与虚拟资源协作是不现实的。许多扩展使用需要同步文件访问和磁盘文件的外部工具。因此，仅支持受限的功能没关系，比如下面列举的的**基础**和**单文件支持**。

A. **基础**语言支持：

- TextMate词元和配色
- 语言特定编辑支持：括号配对、注释、回车规则、折叠标记
- 代码片段

B. **单文件**语言支持：
- 文档符号（概述）、折叠、选择范围
- 文档高亮、语义高亮、文档颜色
- 补全、悬停、签名提示、基于当前文件和静态语言库的符号查找引用/声明
- 格式化、链接编辑
- 词法校验和同文件语义校验和代码操作

C. **跨文件**、**工作空间感知**语言支持：
- 跨文件引用
- 工作空间符号
- 工作空间/项目所有文件校验

以VS Code为载体的富语言扩展（TypeScript、JSON、CSS、HTML、Markdown）在处理虚拟资源时限制于单文件支持

### 禁用语言扩展

如果不能选择在单文件运作，语言扩展也可以决定在虚拟工作空间下禁用扩展。

如果您的扩展同时提供需要禁用的语法和富语言支持，语法也会被禁用。需要避免这种情况，您可以自富语言支持单独创建一个基本语言扩展（语法、语言配置、代码片段），这样就有两个扩展。
- 基本语言扩展设置`"virtualWorkspaces": true`，提供语言ID、配置、语法和代码片段
- 富语言扩展设置`"virtualWorkspaces": false`，包含`main`文件。它贡献语言支持、命令，并且依赖（`expensionDependencies`）于基本语言扩展。富语言扩展应当保留已建立的扩展的扩展ID，这样用户可以在安装单个扩展的情况下拥有完整功能。

您可以在内建语言扩展看到这种方法，如JSON，包含JSON扩展和JSON语言特性扩展。

这种分割同样对在[受限模式](https://code.visualstudio.com/docs/editing/workspaces/workspace-trust#restricted-mode)下运行的[不信任的工作空间](/4.%20扩展指南/13.%20工作空间信任.md)有所帮助。富语言扩展也需要信任，但是基本语言功能可以在任何启动时运作。

### 语言选择器
注册语言特性的提供者时（例如，补全、悬停、代码操作等），确保指定提供者支持的框架：

```typescript
return vscode.languages.registerCompletionItemProvider(
    { language: 'typescript', scheme: 'file' },
    {
        provideCompletionItems(document, position, token) {
            // ...
        }
    }
);
```

这需要添加LSP的文件系统提供器支持。追踪语言服务器协议[issue #1264](https://github.com/microsoft/language-server-protocol/issues/1264)
