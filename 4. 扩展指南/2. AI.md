# VS Code在AI方面的可扩展性
本文提供了Visual Studio Code AI的扩展选择概览，帮助您的扩展选择正确的趋势。

VS Code包含可增强编程体验的强大AI功能：
- **代码补全**：在输入时提供行间代码建议
- **代理模式**：使用特殊工具使AI能够自主规划执行开发任务
- **聊天**：让开发者使用自然语言在聊天界面中，询问有关代码库问题或者对代码库作出编辑
- **智能操作**：处理常见开发任务时使用融合在整个编辑器的AI增强操作

你可以扩展和自定义这些内建功能来创建贴合您的用户特殊需求的定制AI体验。

## 为什么在VS Code里扩展AI

在您的扩展中添加AI功能为您的用户带来一些好处：
- **代理模式下特定领域的知识**：让代理模式访问您公司的数据源和服务
- **增强用户体验**：在您的扩展中提供定制的智能辅助
- **领域专业化**：创建应用于特定编程语言、框架或者领域的AI
- **扩展聊天功能**：在聊天界面中添加特殊工具或者助手带来更强大的互动
- **改良开发者生产力**：使用AI功能推动常规开发者的任务，如调试、代码检查或者测试

## 扩展聊天体验
### 语言模型工具
语言模型工具允许您在VS Code中扩展特定领域能力的代理模式。在代理模式，这些工具基于用户聊天提示词自动调用，并执行特定任务或者从数据源和服务中检索信息。用户也可以明确地在他们的聊天提示词中使用#提及工具引用这些工具。

要实现一个语言模型工具，在您的VS Code扩展中使用[语言模型工具API](/4.%20扩展指南/2.%20AI.md#工具)。语言模型工具可以访问所有的VS Code扩展API，并进一步融入编辑器。

**重要优势**：
- 作为自动编程工作流一部分的专业领域能力
- 您的工具实现可以使用VS Code API，因为它运行在扩展宿主进程
- 在Visual Studio市场中轻易地分发部署

**注意事项**：
- 远程部署需要扩展实现客户端-服务端交流
- 不同工具间的重复利用需要模块化设计和实现

### MCP工具

模型上下文协议（MCP）工具通过使用标准化协议，提供了将外部服务和语言模型的方式。在代理模式，这些工具基于用户聊天提示词自动调用，来处理专业化的任务或者在外部数据源中检索信息。

MCP工具在VS Code外部运行，要么在用户的机器本地运行，或者作为一项远程服务。用户可以通过JSON配置MCP工具，或者让VS Code扩展编程配置他们。你可以通过各种语言SDK和部署选择实现MCP工具。

因为MCP工具在VS Code外部运行，他们无法访问VS Code扩展API。

**重要优势**：
- 作为自动编程工作流一部分的专业领域能力
- 本地远程部署选择
- 在其他MCP客户端重复利用MCP服务端

**注意事项**：
- 无法访问VS Code扩展API
- 分发部署需要用户设置MCP服务器
### 聊天参与者
聊天参与者是允许用户扩展咨询特定领域专家的方式的专业助手。在聊天中，用户可以通过用@提及它唤起聊天参与者，并传递关于特定主题或者领域的自然语言提示词。聊天参与者将负责处理整个聊天互动。

为了实现聊天参与者，在您的VS Code扩展中使用[聊天API](#聊天)。聊天参与者可以访问所有VS Code扩展API，并更加深入融入编辑器。

**重要优势**
- 控制端对端互动流
- 运行在扩展宿主进程中，可以访问VS Code扩展API
- 在Visual Studio市场中轻易分发部署

**注意事项**：
- 远程部署需要扩展实现客户端-服务端交流
- 在不同工具间重复利用需要模块化设计和实现

## 构建您自己的AI赋能特性
为了创建您扩展中自定义的AI赋能特性，VS Code允许您通过编程访问AI模型。这个方法让您能够构建针对编辑器互动的AI能力，而不依赖于聊天界面。

为了直接使用语言模型，在您的VS Code扩展中使用[语言API](#语言模型)。您可以在任何扩展特性中包含这些AI能力，比如代码操作，悬停提供者，自定义视图和其他。

**关键特性**：
- 整合AI能力到已有的扩展特性或构建一个新的。
- 在扩展宿主进程中允许访问VS Code扩展API。
- 通过Visual Studio市场简单的分发和部署

**特别注意：**
- 跨部分的重复利用需要模块化设计和实现

## 作出决定

当在您的VS Code扩展中选择扩展AI的正确方式时，考虑以下指导：
1. 在下面情况选择语言模型工具：
    - 您应该在VS Code中扩展具有特定功能的聊天
    - 您应该在代理模式下基于用户专注的自动化调用
    - 您应该在VS Code中为了深度融合允许访问API
    - 您应该通过VS Code市场分发您的工具
2. 在下面情况选择MCP工具：
    - 您应该在VS Code中扩展具有特定功能的聊天
    - 您应该在代理模式下基于用户专注的自动化调用
    - 您不需要整合VS Code API
    - 您的工具需要在不同环境下工作（不仅仅是VS Code）
    - 您的工具需要远程或本地运行
3. 在下面情况选择聊天参与者：
    - 您应该同通过定制具有该领域专业能力的助手扩展询问模式
    - 您应该自定义整个互动流和反应行为
    - 您应该为了深度融合访问VS Code API
    - 您应该通过VS Code市场分发
4. 在下面情况选择语言模型API
    - 您想要在现有的扩展特性中整合AI能力
    - 您想要在聊天界面之外构建UI体验
    - 您想要直接编程控制AI模型请求

# 语言模型工具API
语言模型工具允许您扩展聊天中大语言模型（LLM）的大部分功能。VSCode中的[代理](https://code.visualstudio.com/docs/copilot/chat/copilot-chat)可以自动调用这些工具，处置聊天当中的特殊任务来用户的聊天提示词。

通过在VS Code扩展中贡献语言模型工具，您可以在扩展代理编程工作流的同时，提供更深层次与编辑器的融合。扩展工具与[内建工具和MCP工具](https://code.visualstudio.com/docs/copilot/chat/chat-tools#_types-of-tools)是三种可用工具类型。

在这篇扩展指南，您可以学习如何通过是使用语言模型工具API创建语言模型工具，和如何实现在聊天扩展中调用的工具。

您也可以通过贡献[MCP服务端](#MCP开发者指南)使用特殊工具扩展聊天体验。关于不同选择和如何确定方法的详情参见[API扩展概述](#vs-code在ai方面的可扩展性)。

> 提示
> 作为终端用户获取更多使用工具的信息，参见[在聊天中使用工具](https://code.visualstudio.com/docs/copilot/chat/chat-tools)

## 什么工具叫做LLM
语言模型工具是可以作为语言模型请求一部分调用的函数。例如，您可能有一个函数，它在数据库中检索信息，处理计算或者调用在线API。当你在VS Code扩展中贡献工具时，代理模式可以在那时基于聊天上下文调用工具。

LLM从不实际执行工具自身，而是生成用于调用工具的参数。清楚描述工具的目的、功能和输入参数是重要的，这样才能让工具可以在正确的上下文被调用。

## 为什么在扩展里实现一个语言模型工具

在实现语言模型工具有几个优势：

- 使用特殊、专业的工具，自动作为用户提示词的回复部分调用，**扩展代理模式**。例如，允许搭建、查询数据库来动态提供LLM相关上下文。

- 使用广泛的扩展API**与VS Code深度融合**。例如，使用[调试API](/4.%20扩展指南/16.%20调试器扩展.md)得到当前的调试上下文并作为工具功能一部分使用。

- 通过Visual Studio市场**分发部署**，提供可依赖和无缝的用户体验。用户不需要单独安装更新工具的过程。

您可能在下面的情景中考虑使用[MCP服务端](#mcp开发者指南)实现语言模型工具：
- 您已经实现了MCP服务端，也想要在VS Code中使用它。
- 您想要在不同的开发环境和平台复用同样的工具。
- 您的工具远程作为宿主服务。
- 您不需要访问VS Code API

## 创建语言模型工具

实现语言模型工具包括两个主要部分：

1. 在扩展的`package.json`文件定义工具配置。
2. 使用[语言模型API引用](/9.%20参考/1.%20VS%20Code%20API.md)在扩展代码中实现工具

您可以使用[基本示例项目](https://github.com/microsoft/vscode-extension-samples/tree/main/chat-sample)开始

### 1. 在package.json中静态配置

定义扩展的语言模型工具的第一步，是在`package.json`文件中定义它。这个配置包括工具名称、简介、输入框架和其他元数据：

1. 在扩展`package.json`文件的`contributes.languageModelTools`字段中为您的工具添加一个项。

2. 给工具一个独特的名称

| 属性 | 简介 |
| - | - |
| `name` | 工具的独特名称，在扩展实现代码中引用。采用`{动词}_{名词}`格式命名。参见[命名指南](#指南与惯例)。 |
| `displayName` | 工具的用户友好名称，在UI中展示 |

3. 如果工具可以使用[代理](#聊天概览)或在聊天提示词中用`#`引用，加入下面属性：

用户可以在聊天视图中启用或禁用工具，类似[模型上下文协议(MCP)工具](https://code.visualstudio.com/docs/copilot/customization/mcp-servers)中所做。

| 属性 | 简介 | 
| - | - |
| `canBeReferencedInPrompt` | 设置为`true`如果工具可以和`代理`一起使用，或可在聊天中引用。 |
| `toolReferenceName` | 用户在聊天上下文中使用`#`引用的名称。 |
| `icon` | 工具在UI中展示的图标。 |
| `userDescription` | 工具的用户友好介绍，在UI中展示。 |

4. 在`modelDescription`中添加详细的介绍。这个信息被LLM用来决定你的工具应该使用哪个上下文。

- 工具实际上做什么
- 它应该返回何种信息
- 何时应当或不应当它被使用
- 介绍工具重要的限制或约束

5. 如果工具需要输入参数，添加`inputSchema`属性介绍工具的输入参数。

这个JSON框架介绍了工具作为输入的属性对象，和它是否应该需要。文件路径应为绝对路径。

介绍各个参数的作用和它与工具功能的关联。

6. 添加`时句`控制工具是否可用

使用`时句`，`languageModelTools`贡献点允许你限制代理模式下提示词中工具可用或可被引用的时间。例如，获取调试调用栈信息的工具，只应在用户调试时可用。

```json
"contributes":{
    "languageModelTools": [
        {
            "name": "chat-tools-sample_tabCount",
            ...
            "when":"debugState == 'running'"
        }
    ]
}
```

### 2. 工具实现
使用[语言模型API](/9.%20参考/1.%20VS%20Code%20API.md)实现语言模型工具。这包括下面步骤

1. 在扩展启用时，使用`vscode.lm.registerTool`注册工具。

如您在`package.json`的`name`属性中指定的那样，提供工具的名称。

如果您想要工具在扩展中私有，跳过工具注册步骤。

```typescript
export function registerChatTools(context: vscode.ExtensionContext) {
    context.subscriptions.push(
        vscode.lm.registerTool('chat-tools-sample_tabCount', new TabCountTool())
    );
}
```

2. 创建实现[`vscode.LanguageModelTool<>`](/9.%20参考/1.%20VS%20Code%20API.md)接口的类

3. 在`prepareInvocation`方法中添加工具确认信息

一个普通的确认对话框会一直在扩展中展示，但工具可以自定义确认信息。给用户足够的上下文理解工具在做什么。这个信息可以是包含代码块的`MarkdownString`。

下面步骤展示了如何为栏数工具提供确认信息。

```typescript
async prepareInvocation(
    options: vscode.LanguageModelToolInvocationPrepareOptions<ITabCountParameters>,
    _token: vscode.CancellationToken
) {
    const confirmationMessages = {
        title: 'Count the number of open tabs',
        message: new vscode.MarkdownString(
            `Count the number of open tabs?` +
            (options.input.tabGroup !== undefined
                ? ` in tab group ${options.input.tabGroup}`
                : '')
        ),
    };
    return {
        invocationMessage: 'Counting the number of tabs',
        confirmationMessages,
    };
}
```

如果`prepareInvocation`返回`undefined`，将展示普通确认信息。注意用户同样可以选择“一直允许”某个工具。

4. 定义描述用户输入参数的接口。

此接口用于`vscode.LanguageModelTool`类的`invoke`方法。输入参数会与您在`package.json`的`inputSchema`定义的JSON框架校对。

下面的粒子展示了栏数工具的接口。

```typescript
export interface ITabCountParameters {
    tabGroup?: number;
}
```

5. 实现`invoke`方法。这个方法在处理聊天提示词调用语言模型工具时被调用。

`invoke`方法接受`options`参数中的工具输入参数。参数会与`package.json`的`inputSchema`定义的JSON框架校对。

当错误发生时，会抛出LLM能理解的错误。您可以选择提供给LLM接下来做什么的说明，比如使用不同参数重试或使用不同的操作。

下面的例子展示了栏数工具的实现。工具的结果是`vscode.LanguageModelToolResult`类型的实例。

```typescript
async invoke(
    options: vscode.LanguageModelToolInvocationOptions<ITabCountParameters>,
    _token: vscode.CancellationToken
) {
    const params = options.input;
    if(typeof params.tabGroup === 'number') {
        const group = vscode.window.tabGroups.all[Math.max(params.tabGroup - 1, 0)];
        const nth = params.tabGroup === 1
            ? '1st'
            : params.tabGroup === 2
                ? '2nd'
                : parmas.tabGroup === 3
                    ? '3rd'
                    : `${params.tabGroup}th`;
        return new vscode.LanguageModelToolResult([new vscode.LanguageModelTextPart(`There are ${group.tabs.length} tabs open in the ${nth} tab group.`)]);

    } else {
        const group = vscode.window.tabGroups.activeTabGroup;
        return new vscode.LanguageModelToolResult([new vscode.LanguageModelTextPart(`There are ${group.tabs.length} tabs open.`)]);
    }
}
```

## 工具调用流程

当用户发送聊天提示词，会发生下面的步骤：

1. Copilot基于用户的配置决定可用的工具列表。

工具列表包括内建工具，扩展注册工具和来自[MCP服务端](https://code.visualstudio.com/docs/copilot/customization/mcp-servers)的工具。您通过扩展或者MCP服务端（图示中用绿色展示）贡献代理模式。

2. Copilot给LLM发送请求，提供提示词、聊天上下文和工具定义列表来思考。

LLM生成回复，可能包括一个或多个请求调用工具。

3. 如果需要，Copilot的LLM提供参数值推荐的工具。

工具回复可能导致多个工具调取请求。

4. 如果有错误或接下来的工具请求，Copilot遍历工具调用流直到所有工具请求解决。

5. Copilot返回最终回复给用户，可能包括不同工具的请求。

### 指南和惯例

- **命名**：为工具和参数写明清晰、简要的名称。
    - **工具名称**：需要独特清晰介绍它的意图。使用`{动词}_{名词}`格式结构为工具命名。例如，`get_weather`，`get_azure_deployment`或`get_terminal_output`。
    - **参数名称**：需要介绍参数的目的。使用`{名词}`格式结构为参数命名。例如：`destination_location`，`ticker`或`file_name`。

- **介绍**：为工具和参数写详细的介绍。
    - 介绍工具做什么和它什么时候应该或不应该使用。例如，“这个工具为给定的位置检索天气”。
    - 介绍每个参数的用法和它与工具功能的关联。例如，“`destination_location`参数指定检索天气的位置。它应该是个合法的位置名称或坐标。”
    - 介绍工具重要的限制或约束。例如，“这个工具仅能为合众国的信息检索天气数据。在其他区域可能无法使用。”

- **用户确认**：提供工具调取的确认信息。虽然普通的确认对话框会在扩展工具中一直展示，但工具可以自定义确认消息。给用户充足的上下文来明白工具在做什么。

- **错误处理**：当错误发生，使用LLM能理解的消息抛出错误。您可以给LLM说明下一步应该做什么，比如使用不同的参数重试或使用不同的操作.

# MCP开发者指南


模型上下文协议（MCP）是让AI模型能通过统一接口与外部工具和服务互动的开放标准。Visual Studio Code实现完全的MCP规范，允许您创建MCP服务端提供工具、提示词和资源扩展VS Code AI代理的能力。

MCP服务端和内建工具、扩展贡献工具同样提供VS Code可用的三种工具类型之一。在[工具类型](https://code.visualstudio.com/docs/copilot/agents/agent-tools#_types-of-tools)了解更多。

本指南涵盖您构建MCP服务端所需要的一切，让您与VS Code和其他MCP客户端紧密协作

> 提示
> 有关作为终端用户使用MCP的信息，参见[在VS Code使用MCP服务端](https://code.visualstudio.com/docs/copilot/customization/mcp-servers)

## 为什么使用MCP服务端

在VS Code中实现MCP服务端扩展语言模型工具聊天有如下优势：

- 使用特殊、专业的工具，自动作为用户提示词的回复部分调用，**扩展代理模式**。例如，允许搭建、查询数据库来动态提供LLM相关上下文。
- 为本地或远程情景提供**灵活部署选择**
- 在不同工具和平台**复用**您的MCP服务端。

您可能会在下列情景考虑用[语言模型API]实现语言模型工具：

- 您想要使用扩展API与VS Code深度融合。
- 您想要通过Visual Studio市场分发工具与更新。

## VS Code支持的MCP特性

VS Code支持下面MCP能力：

- [传输](https://modelcontextprotocol.io/specification/2025-06-18/basic/transports)：
    - 本地标准输入/输出(stdio)
    - 流HTTP(http)
    - 服务端发送事件(sse) - 旧版支持
- [特性](https://modelcontextprotocol.io/specification/2025-06-18#features)：
    - 工具：使用额外工具扩展[代理模式](https://code.visualstudio.com/docs/copilot/chat/copilot-chat#_builtin-chat-modes)
    - 提示词：在聊天使用斜杠命令用复用提示词
    - 资源：提供数据和内容，用户可以添加作为聊天上下文或直接在VS Code中互动。
    - 获取：向用户请求输入
    - 采样：使语言模型请求使用用户配置模型和订阅
    - 验证：使用OAuth验证对MCP服务端的访问
    - 服务端说明
    - 根：提供用户工作空间根目录的信息
    - [MCP应用程序](https://modelcontextprotocol.github.io/ext-apps/api/)：返回工具互动性UI组件
### 工具
#### 工具定义
VS Code支持代理模式下的MCP工具。在此模式下，MCP工具可以基于任务需要调用。用户可以使用工具选择器启用或配置它们。工具简介在工具选择器以及在运行工具前的确认窗口和工具名称一同展示。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-tools-picker.png)

用户可以编辑在工具确认对话框中的模型生成的输入参数。确认对话框会为所有没有被标记为`readOnlyHint`注解的工具展示。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-tool-input-parameters.png)

#### 动态工具搜索
VS Code也支持[动态工具搜索](https://modelcontextprotocol.io/specification/2025-06-18/server/tools#tool-discovery-and-updates)，允许服务端在运行时注册工具。例如，服务端可以基于检测到工作空间或用户聊天提示词的语言或框架提供不同的工具。

#### 工具注解
为了提供额外关于工具行为元数据，您可以使用[工具注解](https://modelcontextprotocol.io/docs/concepts/tools#tool-annotations)
- `title`：人类可阅读的工具标题，当工具被调用时在聊天视图展示。
- `readOnlyHint`：表明工具只读的可选提示。VS Code不会在运行只读工具前征求同意。

### 资源
资源允许您用结构化的方式提供数据和内容给用户。用户可以直接在VS Code中访问资源，或者在聊天提示词中用作上下文。例如，MCP服务端可以生成截图，使它们作为资源，或提供对实时更新的日志文件的访问。

当您定义MCP资源时，资源名称在MCP资源快速选取中展示。资源可以通过**MCP：浏览资源**命令或在聊天请求中找到**添加上下文**，然后选择**MCP资源**。资源可以包括文本或二进制内容。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-resources-picker.png)

VS Code支持资源更新，允许用户在编辑器中实时查看资源内容更新。

#### 资源模板

VS Code同样支持[资源模板](https://modelcontextprotocol.io/specification/2025-06-18/server/resources#resource-templates)，允许用户在引用资源时提供输入参数。例如，数据库查询工具可能要求数据表名称。

当访问模板资源时，有哦那个在快速选单中提供需要的参数。您可以提供参数建议值的补全。

### 提示词
在用户可以通过斜杠命令(`mcp.servername.promptname`)调用聊天提示词模板的聊天中，提示词是可以复用的。提示词在引导用户使用您的服务端是非常有用的，它可以强调不同的工具或提供内建复杂工作流以适应用户本地上下文和服务。

如果您定义了提示词输入参数建议值的[补全](https://modelcontextprotocol.io/specification/2025-06-18/server/utilities/completion)，VS Code会显示对话框自用户收集输入。

```typescript
server.prompt(
    'teamGreeting',
    'Generate a greeting for team members',
    {
        name: completable(z.string(), value => {
            return ['Alice', 'Bob', 'Charlie'].filter(n => n.startsWith(value));
        })
    },
    async ({ name } => ({
        messages: [
            {
                role: 'assistant',
                content: { type: 'text', text: `Hello ${name}, welcome to the team!` }
            }
        ]
    }))
);
```
![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-prompt-argument.png)

> 注意
> 用户可以在提示词对话框输入控制台命令并使用命令输出作为提示词的输入。

当您的提示词响应包含资源类型时，VS Code将资源作为上下文提供给聊天提示词。

### 验证
VS Code提供的MCP服务端需要验证，允许用户与一个代表它们服务账户的MCP服务端互动。

[验证规范](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization)清晰地将MCP服务端分割为资源服务端和验证服务端，允许开发者将验证委托给已有的身份认证提供者（IdP）而非从零构建自己的OAuth实现。

VS Code有来自GitHub和Microsoft Entra的内建验证支持。如果您的MCP服务端实现最新的规范并且使用GitHub或Microsoft Entra作为验证服务端，用户可以通过**账户菜单** > **管理信任的MCP服务端**操作管理哪些MCP服务端可以访问账户。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/manage-trusted-mcp.png)

VS Code支持使用GitHub和Microsoft Entra之外的其他OAuth 2.1标准和2.0标准验证。VS Code会启动[动态客户端注册（DCR）](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization#dynamic-client-registration)握手，如果IdP不支持DCR，随后回到客户端证书工作流。这为每个MCP服务端不同的IdP或特殊客户端ID秘密对有根据地提供了更多灵活性。

用户随后可以通过**账户菜单**查看验证状态。如果要移除动态客户端注册，用户可以在命令面板使用**验证：移除动态验证提供者**命令。

在下面是确保MCP服务端和VS Code OAuth工作流正常工作的检查列表：

1. MCP服务端定义了[MCP验证规范](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization)
2. IdP必须支持DCR或客户端证书中的一种。

3. URL重定向列表必须包含这些URL：`http://127.0.0.1:33418`和`https://vscode.dev/redirect`

当DCR不被MCP服务端支持时，用户会回到客户端证书流程：
![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-auth-dynamic-client-required.png)
![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-auth-client-id.png)
![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-auth-client-secret.png)
> 注意
> VS Code仍然支持作为验证服务端的MCP服务端，但推荐为最新服务端使用最新规范。

### 采样

VS Code为MCP服务端提供了[采样](https://modelcontextprotocol.io/specification/2025-06-18/client/sampling)权限。这允许您的MCP服务端使用用户配置模型和订阅发出语言模型请求。例如，使用采样总结大数据集，在发送给客户端前提取信息，或在工具中实现代理决策逻辑。

MCP首次使用采样请求，用户会提示授权服务端访问模型。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-allow-sampling.png)

当特定模型发出采样请求时，考虑用户可以限制MCP服务端能够使用命令面板中的**MCP：服务端列表 -> 配置模型访问**命令使用哪些模型。当你在MCP服务端指定`modelPreferences`提供关于使用采样的模型提示时，VS Code会从允许的模型中挑选。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/mcp/mcp-configure-model-access.png)
用户可以在命令面板中，使用**MCP：服务端列表 > 展示采样请求**命令查看MCP服务端发出的采样请求。

### 工作空间根
VS Code可以将用户工作空间根目录信息提供给MCP服务端。

### MCP应用程序
MCP应用程序让工具可以返回互动式UI组件，它们能在行间渲染出来而不仅仅是文本输出。这对拖拽列表排序、可视化、表单和多步骤工作流等场景很实用。

#### 架构

MCP应用程序使用工具+UI资源形式：
1. 定义返回一个指向UI资源的`_meta.ui.resourceUri`的工具
2. 使用`ui://`URI框架和MIME类型`text/html;profile=mcp-app`创建UI资源
3. HTML资源在沙盒内联框架中运行，并且使用MCP应用程序SDK与VS Code交流
#### SDK
使用[@modelcontextprotocol/ext-apps](https://github.com/modelcontextprotocol/ext-apps)包构建MCP应用程序。SDK提供：

- 应用程序类：与宿主沟通的主要接口
    - `connet()`：与VS Code建立连接
    - `callServerTool(name, args)`：在建立MCP服务端时调用工具
    - `sendMessage(content)`：给聊天输入发送消息
    - `updateModelContext(context)`：给未来聊天交换提供上下文
    - `openLink(url)`：请求在浏览器打开URL
    - `sendLog(level, message)`：发送调试日志（不加入聊天）
- 通知处理器：设置以从VS Code接收事件
    - `ontoolinput`：接收完整工具参数
    - `ontoolinputpartial`：接收部分流输入
    - `ontoolresult`：接收工具执行结果
    - `ontoolcancelled`：处理工具撤销任务
    - `onhostcontextchanged`：对主题或地点变化反应
    - `onteardown`：在卸载时清理
#### VS Code特点与限制
| 特性 | VS Code支持 |
| - | - |
| 显示模式 | 仅`inline`（不支持`fullscreen`或`pip`）|
| 发送消息 | 填充聊天输入框；不能自动发送 |
| 上下文更新 | 作为附属 |
| 剪切板写入 | 支持 |
| 摄像头、麦克风、地理位置 | 不支持 |
#### 安全
MCP应用程序运行在沙盒内联框架，强制执行内容安全协议（CSP）。当定义UI资源时，声明您的应用程序需要访问的域：
- `connectDomains`：fetch/XHR请求域
- `resourceDomains`：图像、字体和其他资源域
- `frameDomains`：可嵌入内联框架的域

### 图标
VS Code支持MCP服务端、资源和工具提供`icons`。MCP图标具有`src`属性，作为图像的URI：
- 使用HTTP或SSE通信的MCP服务端可以提供这样的图像，这些图像来自托管MCP的同一来源。例如，配置于`https://example.com/mcp`的服务端可能提供来自`example.com`的图像。
- MCP服务端使用stdio通信可以使用`file:///`URI提供来自文件系统的图像。
- 任何MCP服务端可以使用`data:`将图像作为数据URI嵌入。
## 给VS Code添加MCP服务端
用户在VS Code可以通过以下几种方式添加MCP服务端：

- 直接从网络安装：在网站上使用特定MCP安装URL（`vscode:mcp/install`）。
- 工作空间配置：在工作空间中使用`.vscode/mcp.json`文件指定服务端配置。
- 全局配置：在用户[档案](https://code.visualstudio.com/docs/configure/profiles)中进行全局服务端定义。
- 自动发现：VS Code可以从其他类似Claude Desktop的工具发现服务端。
- 扩展：VS Code扩展可以程序化注册MCP服务端。
- 命令行：使用`--add-mcp`VS Code命令行选项安装MCP服务端。

了解更多[给VS Code添加MCP服务端](https://code.visualstudio.com/docs/copilot/customization/mcp-servers#add-an-mcp-server)的方式。

## 从安装URL创建MCP
VS Code提供URL处理器从链接安装MCP服务端：`vscode:mcp/install?{json-configuration}`（Insider：`vscode-insider:mcp/install?{json-configuration}`）

使用下面格式提供JSON服务端配置：`{\"name\":\"server-name\", \"command\":...}`随后进行JSON字符串转换和URL解码。例如，使用下面逻辑创建安装URL：

```typescript
//对于Insider，使用`vscode-insiders`替代`code`
const link = `vscode:mcp/install?${encodeURIComponent(JSON.stringify(obj))}`;
```

链接可以在浏览器使用，或在命令行打开，例如在Linusx通过`xdg-open $LINK`

## 在扩展中注册MCP服务端
要在扩展中注册MCP服务端，您需要进行以下步骤：
1. 在扩展的`package.json`文件中定义MCP服务端定义提供者
2. 在扩展代码中，使用[vscode.lm.registerMcpServerDefinitionProvider](/9.%20参考/1.%20VS%20Code%20API.md)API实现MCP服务端定义提供者。

您可以从基本的[在VS Code扩展中注册MCP服务端示例](https://github.com/microsoft/vscode-extension-samples/blob/main/mcp-extension-sample)开始

### 1. 在package.json中静态配置

想要注册MCP服务端的扩展，必须在`package.json`贡献`contributes.mcpServerDefinitionProviders`扩展点，提供提供者的`id`。这个`id`应当与实现中的匹配。
```json
{
    ...
    "contributes": {
        "mcpServerDefinitionProviders": [
            {
                "id": "exampleProvider",
                "label": "Example MCP Server Provider"
            }
        ]
    }
    ...
}
```

### 2.实现提供者
要在扩展中注册MCP服务端，使用[`vscode.lm.registerMcpServerDefinitionProvider`](/9.%20参考/1.%20VS%20Code%20API.md)API提供服务端的[MCP配置](https://code.visualstudio.com/docs/copilot/chat/mcp-servers#_configuration-format)。API需要`providerId`字符串和`McpServerDefinitionProvider`对象。

`McpServerDefinitionProvider`对象具有三个属性：

- `onDidChangeMcpServerDefinition`：MCP服务端配置改变的事件。
- `provideMcpServerDefinitions`：返回MCP服务端配置数组（`vscode.McpServerDefinition[]`）的函数。
- `resolveMcpServerDefinition`：MCP服务端需要启动时编辑器调用的函数。使用此函数进行额外需要用户互动的操作，比如授权。

`McpServerDefinition`对象可以是以下类型之一：
- `vscode.McpStdioServerDefinition`：代表MCP服务端可在运行本地进程、用标准输入输出流操作时使用。
- `vscode.McpHttpServerDefinition`：代表MCP服务端可在流式HTTP传输使用。

<details>
<summary>
<b>
示例MCP服务端定义提供者
</b>
</summary>
下面的示例展示了如何在扩展中注册MCP服务端，在启动服务端时请求用户输入API密匙

```typescript
import * as vscode from 'vscode';

export function activate(context: vscode.ExtensionContext) {
    const didChangeEmitter = new vscode.EventEmitter<void>();

    context.subscriptions.push(vscode.lm.registerMcpServerDefinitionProvider('exampleProvider', {
        onDidChangeMcpServerDefinitions: didChangeEmitter.event,
        provideMcpServerDefinitions: async () => {
            let servers: vscode.McpServerDefinition[] = [];

            //简单标准输入输出服务端定义示例
            servers.push(new vscode.McpStdioServerDefinition(
            {
                label: 'myServer',
                command: 'node',,
                args: ['server.js'],
                cwd: vscode.Uri.file('/path/to/server'),
                env: {
                    API_KEY: ''
                },
                version: '1.0.0'
            });

            //HTTP服务端定义示例
            servers.push(new vscode.McpHttpServerDefinition(
            {
                label: 'myRemoteServer',
                uri: 'http://localhost:3000',
                headers: {
                    'API_VERSION': '1.0.0'
                },
                version: '1.0.0'
            }));
            return servers;
        },
        resolveMcpServerDefinition: async (server: vscode.McpServerDefinition) => {
            if(server.label == 'myServer') {
                //自用户获取API密匙，例如使用vscode.window.showInputBox
                //使用API密匙更新服务端定义
            }
            //返回undefined表明服务端不应该启动或抛出了错误
            //如果有待处理的工具调用，编辑器会取消并返回错误信息
            //给语言模型
            return server;
        }
    }));
}
```
</details>

## 修正错误和调试MCP服务端
### VS Code的MCP开发模式
当开发MCP服务端时，通过为服务端配置添加`dev`键，您可以为MCP服务端启用*开发模式*。这个对象有两个属性:
- `watch`：一个用于监听可能会重启MCP服务端文件变化的文件通配符。
- `debug`：使您能够与MCP服务端一同启动调试器。当前，VS Code支持调试Node.js和Python MCP服务端

<details>
<summary>
<b>
Node.js MCP服务端
</b>
</summary>

要调试Node.js MCP服务端，设置`debug.type`属性为`node`。
```json
{
    "servers": {
        "my-mcp-server": {
            "type": "stdio",
            "command": "node",
            "cwd": "${workspaceFolder}",
            "args": ["./build/index.js"],
            "dev": {
                "watch": "src/**/*.ts",
                "debug": { "type": "node" }
            }
        }
    }
}
```
</details>

{
    "servers": {

# 聊天参与者API

聊天参与者是允许用户扩展[VS Code聊天](https://code.visualstudio.com/docs/copilot/chat/copilot-chat)的定制化助理，它们是各领域的专家。用户通过@提及调用聊天参与者，聊天参与者随后会回复处理用户自然语言提示词。

在本扩展指南，您会学习如何通过聊天参与者API创建聊天参与者。

VS Code拥有几个内建聊天参与者，如`@vscode`、`@terminal`和`@workspace`。它们的回答专业领域问题的能力得到了优化。

聊天参与者与[语言模型工具](#语言模型工具)不同，语言模型工具是作为LLM一部分调用组织起对用户聊天提示词的解决步骤。聊天参与者接收用户提示词，组织需要它们自己需要的任务。

## 为什么在扩展中实现聊天参与者

在扩展中实现聊天参与者有几个优势：
- 用专业、专门的知识与技能**扩展聊天**。例如，内建`@vscode`参与者对VS Code和扩展API了如指掌。

- 管理端对端用户聊天提示词与回复**主导对话**。
- 广泛使用扩展API**与VS Code深度融合**。例如，使用[调试API](/4.%20扩展指南/16.%20调试器扩展.md)获得当前的调试器上下文，并将其作为工具一部分。
- 通过Visual Studio市场**分发部署**聊天参与者，提供可靠、无缝的体验给用户。用户不需要额外安装和更新过程。

如果您想要提供自主代理编程时段的能力，您也许会考虑实现[语言模型工具](#语言模型工具)或者[MCP服务端](#mcp工具)。参见[AI可扩展性概览]获得不同方式和如何决定的细节。

## 创建聊天参与者
实现聊天参与者由下面步骤组成：
1. 在扩展`package.json`文件中定义聊天参与者
2. 实现请求处理器处理用户聊天提示词并返回回复。
3. （可选）实现聊天斜杠命令提供用户通用任务的快捷标记。
4. （可选）定义接续问题。
5. （可选）实现参与者推断，让VS Code自动规划聊天请求给合适的参与者，而不需要让用户分心。

您可以从[基本示例项目](https://github.com/microsoft/vscode-extension-samples/tree/main/chat-sample)开始。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/chat/diagram.png)

### 1.注册聊天参与者
创建聊天参与者的第一步是在`package.json`中注册，包含下面属性：
- `id`：聊天参与者独特的标识符，在`package.json`文件中定义。
- `name`：聊天参与者简短的名称，在聊天中用@提及。
- `fullName`：聊天参与者的全名，在回复标题区域显示。
- `description`：聊天参与者目的的简短介绍，用于在聊天输入框中占位。
- `isSticky`：表明聊天参与者是否在回复后一直处于聊天输入框中的布尔值。

```json
"contributes": {
    "chatParticipants": [
        {
            "id": "chat-sample.my-participant",
            "name": "my-participant",
            "fullName": "My Participant",
            "description": "What can I teach you?",
            "isSticky": true
        }
    ]
}
```

我们建议使用小写`name`和使用标题式`fullName`来与已有的聊天参与者统一。在[聊天参与者命名规范](#聊天参与者命名规范)

> 注意
> 一些参与者名称被保留。如果您使用被保留的名称，VS Code会显示聊天参与者的完整合格名称（包括扩展ID）。

### 2. 实现请求处理器

使用[聊天参与者API](/9.%20参考/1.%20VS%20Code%20API.md)实现请聊天参与者。这包括如下步骤：

1. 在扩展启用时，使用`vscode.chat.createChatParticipant`创建参与者。

提供您在`package.json`中定义的ID和您在下一步实现的请求处理器的引用。

```typescript
export function activate(context: vscode.ExtensionContext) {
    //注册聊天参与者和请求处理器
    const cat = vscode.chat.createChatParticipant('chat-sample.my-participant', handler);

    //可以为@cat设置一些属性
    cat.iconPath = vscode.Uri.joinPath(context.extensionUri, 'cat.jpeg');

    //在此添加聊天请求处理器
}
```

2. 在`activate`函数，定义`vscode.ChatRequestHandler`请求处理器。

请求处理器负责在VS Code聊天视图中处理用户聊天请求。每次用户在聊天输入框中输入提示词，就会调用聊天处理器。

```typescript
const handler: vscode.ChatRequestHandler = async (
    request: vscode.ChatRequest,
    context: vscode.ChatContext,
    stream: vscode.ChatResponseStream,
    token: vscode.CancellationToken
): Promise<ICatChatResult> => {
    // 这里是聊天请求处理器的实现
};
```
3. 判断来自`vscode.ChatRequest`的用户意图。

为了判断用户请求的意图，您可以引用`vscode.ChatRequest`参数访问用户提示词上下文、命令和聊天位置。

您可以选择利用语言模型判断用户意图，而不是使用传统逻辑。作为`request`对象的一部分，您会得到用户在聊天模型下拉菜单中选择的语言模型实例。了解使用[语言模型API](#语言模型工具)的方法。

下面的代码片段展示了首先使用命令，随后通过用户提示词判断用户意图的基本结构：

```typescript
const handler: vscode.ChatRequestHandler = async (
    request: vscode.ChatRequest,
    context: vscode.ChatContext,
    stream: vscode.ChatResponseStream,
    token: vscode.CancellationToken
): Promise<ICatChatResult> => {
    //检查`teach`命令
    if(request.command == 'teach') {
        //在此添加逻辑处理教学场景
        doTeaching(request.prompt, request.variables);
    } else {
        //判断用户意图
        const intent = determineUserIntent(request.prompt, request.variables, request.model);
        
        //在此添加逻辑处理其他场景
    }
}
```

4. 添加逻辑处理用户请求。

通常聊天扩展使用`request.model`语言模型实例处理请求，在这种情况下您会调整语言模型提示词匹配用户意图。

或者，您可以调用后端服务、使用传统编程逻辑或结合以上实现额外的逻辑。例如，您可以调用网络搜索汇集额外信息，提供作为语言模型上下文。

在处理当前请求，您可能会想参考之前的聊天信息。例如，如果之前的回复返回C#代码片段，用户当前的请求也许是“给我Python代码”。[了解如何使用聊天信息历史](#使用聊天信息历史)。

如果您想要基于完全不同的聊天输入位置（聊天视图，快速聊天，内联聊天）处理请求，您可以使用`vscode.ChatRequest`的`location`实行。例如，如果从终端内联聊天发送请求，您可能会查阅命令行命令。然而，如果用户使用聊天视图，您会返回更复杂的回复。

5. 返回聊天回复给用户。

一旦您处理完了请求，您必须在聊天视图将回复返回给用户。您可以使用流回复用户查询。

回复可以包含不同的内容类型：Markdown、图像、引用、进度条、按钮和文件树。

![](https://code.visualstudio.com/assets/api/extension-guides/ai/chat/stream.png)

扩展可以用下面方式使用回复流：

```typescript
stream.progress('Picking the right topic to teach...');
stream.markdown(`\`\`\` typescript
const myStack = new Stack();
myStack.push(1); //将一个数字入栈（或者说，让一条鱼入栈）
myStack.push(2); //加入另一条鱼（数字2）
console.log(myStack.pop()); // 吃最上面的鱼，会输出：2
\`\`\`
所以记住，在栈中，编程猫，最后的鱼先出，我们会教猫说后入先出（最后进去的，首先出来）。`);

stream.button({
    command: `cat.meow`,
    title: vscode.l10n.t('Meow!'),
    arguments: []
});
```

了解更多关于[支持的聊天回复输出类型](#支持的聊天回复输出类型)信息。

在实践中，扩展一般会发送请求给语言模型。一旦它们从语言模型得到回复，它们会进一步处理，然后决定是否流传输回给用户什么。VS Code聊天API是基于流的，并且和流式[语言模型API](#语言模型工具api)兼容。这允许扩展持续报告进度和结果，以得到流畅的用户体验。了解如何使用[语言模型API](#语言模型工具api)

### 4. 注册接续请求
在每次聊天请求之后，VS Code调用接续提供者获取建议的接续问题，并显示给用户。用户可以选择接续问题，并立即发送给聊天扩展。使用[`ChatFollowupProvider`](/9.%20参考/1.%20VS%20Code%20API.md)API注册[`ChatFollowup`](/9.%20参考/1.%20VS%20Code%20API.md)类型的接续提示词。

下面的代码片段展示了如何在聊天扩展中注册接续请求：

```typescript
cat.followupProvider = {
    provideFollowups(result: ICatChatResult, context: vscode.ChatContext, token: vscode.CancellationToken) {
        if(result.metadata.command === 'teach') {
            return [{
                prompt: 'let us play',
                label: vscode.l10n.t('Play with the cat')
            } satisfies vscode.ChatFollowup];
        }
    }
}
```
> 提示
> 接续应当为问题或指引，而不仅仅是简介的命令。

### 5. 实现参与者推断

为了更简单地用自然语言使用聊天参与者，您可以实现参与者推断。参与者推断是自动安排用户问题给合适的参与者的方法，而不需要单独在提示词中提及参与者。例如，如果用户问：“我怎么样在项目中添加登录页面？”，问题会自动分配给`@workspace`参与者，因为它可以回答关于用户项目的问题。

VS Code使用聊天参与者简介和示例决定分配聊天提示词给哪个参与者。您可以在扩展`package.json`文件中的`disambiguation`属性中指定该信息。`disambiguation`属性包括一系列推断分类，每个都有简介和示例。

| 属性 | 简介 | 示例 |
| - | - | - |
| `category` | 推断分类。如果参与者有不同的目的，您可以给每个分配一个分类。 | <ul><li>`cat`</li><li>`workspace_question`</li><li>`web_question`</li></ul> |
| `description` | 适合该参与者的问题是什么样 | <ul><li>`用户想要用非正式的方式了解特定的计算机科学主题`</li><li>`用户只是想要放松，看看猫玩耍`</li></ul> |
| `examples` | 代表性的问题示例列表 |<ul><li>`用比喻教我C++指针`</li><li>`用简单的方式给我解释什么是链表`</li><li>`您可以给我看看猫怎么和激光笔玩吗？`</li></ul> |

您可以定义参与者推断，使之成为全面的聊天参与者，或者是用于特别命令，或者是两者的结合。

下面的代码片段展示了如何在参与者层实现参与者推断。

```json
"contributes": {
    "chatParticipants": [
        {
            "id": "chat-sample.cat",
            "fullName": "Cat",
            "name": "cat",
            "description": "喵！我可以教你什么？",

            "disambiguation": [
                {
                    "category": "cat",
                    "description": "用户想要用非正式的方式了解特定的计算机科学主题",
                    "examples": [
                        "用比喻教我C++指针",
                        "用简单的方式给我解释什么是链表",
                        "你可以解释下编程中的函数是什么吗？"
                    ]
                }
            ]
        }
    ]
}
```

类似的，您也可以在命令层的`commands`属性中，通过添加具有一个或多个项目的`disambiguation`属性配置参与者推断。

应用下面的导引改善扩展参与者的精确度：

- **专一**：简介和示例应当尽可能专业，避免与其他参与者的冲突。避免在参与者和命令信息中使用宽泛的术语。
- **使用示例**：示例应能代表适合本参与者的问题类型。使用同义词和变化词汇涵盖广泛的用户需求。
- **使用自然语言**：既然您正在给用户解释参与者，简介和示例应当使用自然语言书写。
- **测试推断**：使用不同的示例问题测试参与者推断，验证没有与内建聊天参与者的冲突。

> 注意
> 内建聊天参与者优于参与者推断。例如，操作工作空间文件的聊天参与者会与内建`workspace`参与者冲突。

## 使用聊天信息历史

参与者能够访问当前聊天段的信息历史。参与者仅能访问它被提及的信息。`history`项目为`ChatRequestTurn`或者`ChatResponseReturn`。例如，使用下面的代码片段检索所有之前当前聊天段内用户发送给参与者的请求。

```typescript
const previousMessages = context.history.filter(h => instanceof vscode.ChatRequestTurn);
```

历史不会被自动包括在提示词内，这取决于参与者是否希望将历史作为传递信息给语言模型时额外的上下文。

## 支持的聊天回复输出类型

给聊天请求返回回复，您可以在[`ChatRequestHandler`](/9.%20参考/1.%20VS%20Code%20API.md)中使用[`ChatResponseStream`](/9.%20参考/1.%20VS%20Code%20API.md)参数。

下面的列表提供了聊天视图中的输出类型。聊天回复可以结合多种不同的输出类型。

- **Markdown**

渲染Markdown简单文本或图像的文本段。您可以使用[CommonMark](https://commonmark.org/)规范的任何Markdown语法。使用[`ChatResponseStream.markdown`](/9.%20参考/1.%20VS%20Code%20API.md)方法提供Markdown文本。

示例代码片段：

```typescript
// 渲染Markdown文本
stream.markdown('# 这是标题\n');
stream.markdown('这是格式化文本，使用_斜体_和**粗体**。');
stream.markdown('这是一个[链接](https://code.visualstudio.com)。\n\n');
stream.markdown(' ![VS Code](https://code.visualstudio.com/asset/favicon.ico)');
```

- 代码块
渲染支持智能感知、代码格式化和互动控制的代码块，并应用于活跃的编辑器。为了显示代码块，使用[ChatResponseStream.markdown]方法，并使用Markdown代码块语法（使用反引号）。

示例代码片段：
```typescript
// 渲染允许用户互动的代码块
stream.markdown('```bash\n');
stream.markdown('```ls -l\n');
stream.markdown('```');
```

- **命令链接**
在聊天回复中渲染用户可以选择并调用VS Code命令的内联链接。使用[`ChatResponseStream.markdown`](/9.%20参考/1.%20VS%20Code%20API.md)方法，使用Markdown链接语法`[链接文本](command:commandId)`，在此URL中提供命令ID以显示命令链接。例如，下面的链接打开命令面板：`[命令面板](command:workbench.action.showCommands)`。

在您从服务中加载Markdown文本时，要防止命令注入，您必须使用[`vscode.MarkdownString`]对象，其`isTrusted`属性设置为信任的VS Code命令ID列表。使命令链接运作需要这个属性。如果`isTrusted`属性没有设置或命令不在列表中，命令链接不会正常运作。

示例代码片段：


```typescript
// 使用命令URI从Markdown链接命令
let markdownCommandString: vscode.MarkdownString = new vscode.MarkdownString(
    `[使用猫名称](command:${CAT_NAMES_COMMAND_ID})`
);
markdownCommandString.isTrusted = { enabledCommands: [CAT_NAMES_COMMAND_ID] };

stream.markdown(markdownCommandString);
```

如果命令需要参数，您需要一个JSON编码参数，随后编码JSON字符串作为URI组件。随后添加编码后的参数作为命令链接的查询字符串。

```typescript
//编码命令参数
const encodedArgs = encodeURIComponent(JSON.stringify(args));

// 使用带参数的命令URI从Markdown链接命令
let markdownCommandString: vscode.MarkdownString = new vscode.MarkdownString(
    `[使用猫名称](command:${CAT_NAMES_COMMAND_ID}?${encodedArgs})`
);
markdownCommandString.isTrusted = {enabledCommands: [CAT_NAMES_COMMAND_ID] };
stream.markdown(markdownCommandString);
```

- **命令按钮**

渲染能调用VS Code命令的按钮。命令可以是内建的也可以是您在扩展中定义的。使用[`ChatResponseStream.button`](/9.%20参考/1.%20VS%20Code%20API.md)方法，提供按钮文本和命令ID

示例代码片段：
```typescript
// 渲染按钮触发VS Code命令
stream.button({
    command: 'my.command',
    title: vscode.l10n.t('运行我的命令')
});
```

- **文件树**
渲染文件树，让玩家能够预览单独预览文件。例如，想要创建新工作空间使显示工作空间预览。使用[`ChatResponse.Stream.filetree`](/9.%20参考/1.%20VS%20Code%20API.md)方法，提供文件树元素的数组和文件的根位置（文件夹）。

示例代码片段：

```typescript
// 创建文件树实例
var tree: vscode.ChatResponseFileTree[] = [
    {
        name: 'myworkspace',
        children: [{ name: 'README'}, { name: 'app.js' }, { name: 'package.json' }]
    }
];

// 在根位置渲染文件树控制器
stream.filetree(tree, baseLocation);
```

- **进度信息**
在一个长期运行操作中渲染进度信息，提供用户中途的反馈。例如，报告多步骤操作中每一步的完成情况。使用[`ChatResponseStream.progress`](/9.%20参考/1.%20VS%20Code%20API.md)方法，提供消息。

实例代码片段：

```typescript
// 渲染进度消息
stream.progress('正在连接数据库。');
```

- **引用**
在引用列表中添加外部URL或编辑器位置的引用，指出您用作上下文的信息。使用[`ChatResponseStream.reference`](/9.%20参考/1.%20VS%20Code%20API.md)方法提供引用位置。

实例代码片段：

```typescript
const fileUri: vscode.Uri = vscode.Uri.file('/path/to/workspace/app.js'); 
const fileRange: vscode.Range = new vscode.Range(0, 0, 3, 0);
const externalUri: vscode.Uri = vscode.Uri.parse('https://code.visualstudio.com');

// 添加对整个文件的引用
stream.reference(fileUri);

// 在文件内添加对特定段落的引用
stream.reference(new vscode.Location(fileUri, fileRange));

// 添加对外部链接的引用

stream.reference(externalUri);
```

- **内联引用**

在URI或编辑器位置中添加内联引用。使用[`ChatResponseStream.anchor`](/9.%20参考/1.%20VS%20Code%20API.md)方法，并提供锚点位置和可选的标题。要引用某符号（例如，类或者变量），您将使用编辑器中的位置。

实例代码片段：

```typescript
const symbolLocation: vscode.Uri = vscode.Uri.parse('location-to-a-symbol');

//渲染工作空间内符号的内联锚点
stream.anchor(symbolLocation, 'MySymbol');
```

**重点：**图像和链接仅在源自信任的域列表中的范围可用。了解更多关于[VS Code链接保护](https://code.visualstudio.com/docs/editing/editingevolved#outgoing-link-protection)的信息。

## 实现工具调用

要对用户请求作出回复，聊天扩展可以调用语言模型工具。了解更多关于[语言模型工具](#语言模型工具api)和[工具调用流](#工具调用流)。

您可以用两种方法实现工具调用：

- 使用[`@vscode/chat-extension-utils`库](https://www.npmjs.com/package/@vscode/chat-extension-utils)简化在聊天扩展中调用工具的过程。
- 自己实现工具调用，这会带来对工具调用过程更全面的控制。例如：在发送给LLM之前，要用特别方式实现额外的验证或者处理工具回复。

### 使用聊天扩展库实现工具调用
您可以使用[`@vscode/chat-extension-utils`库](https://www.npmjs.com/package/@vscode/chat-extension-utils)简化聊天扩展中调用工具的过程。

在聊天参与者的`vscode.ChatRequestHandler`函数中实现工具调用。
1. 判断当前聊天上下文的相关工具。您可以通过`vscode.lm.tools`访问所有可用的工具。

下面的代码片段展示了如何筛选拥有特定标签的工具。

```typescript
const tools = 
    request.command === 'all'
        ? vscode.lm.tools
        : vscode.lm.tools.filter(tool => tool.tags.includes('chat-tools-sample'));
```

2. 使用`sendChatParticipantRequest`发送请求和哦概念股据定义给LLM。

```typescript
const libResult = chatUtils.sendChatParticipantRequest(
    request,
    chatContext,
    {
        prompt: '你是一只猫！像只猫一样回复。',
        responseStreamOptions: {
            stream,
            references: true,
            responseText:true
        },
        tools
    },
    token
);
```

`ChatHandlerOptions`对象具有下面的属性：
- `prompt`：（可选）说明聊天参与者的提示词。
- `model`：（可选）请求使用的模型。如果未指定，会使用来自聊天上下文的模型。
- `tools`：（可选）请求需考虑的工具列表。
- `requestJustification`：（可选）描述请求为什么会发出的字符串。
- `responseStreamOptions`：（可选）启用`sendChatParticipantRequest`流式传输回复给VS Code。您可以启用引用和回复文本。

3. 从LLM返回结果。可能包括错误详情或工具调用元数据
```typescript
return await libResult.result;
```

完整的[工具调用实例](https://github.com/microsoft/vscode-extension-samples/blob/main/chat-sample/src/chatUtilsSample.ts)源代码在VS Code扩展实例仓库可用。

### 自己实现工具调用

对于更高阶的情景，您可以自己实现工具调用。或者，使用`@vscode/prompt-tsx`库撰写LLM提示词。通过自己实现工具调用，您可以对整个工具调用过程有更充分的控制。例如，在发送给LLM之前，用特别的方式实现额外的验证或者处理工具回复。

在VS Code扩展示例仓库中查看完整实现[使用prompt-tsx实现工具调用](https://github.com/microsoft/vscode-extension-samples/blob/main/chat-sample/src/toolParticipant.ts)的完整源代码。

我们推荐您添加远程记录`无用`用户反馈事件和参与者处理的请求总数来评估成果。初始的参与者成功度可以被定义为：`unhelpful_feedback_count / total_requests`。

```typescript
const logger = vscode.env.createTelemetryLogger({
    //远程记录实现在这

任何其他与聊天回复的用户互动应该被作为正向度量计算（例如，用户选择了生成在聊天回复中的按钮）。即便使用远程测量成功度并不是决定性技术，但也是与AI协作的重要部分。运行测验，评估，反复改进您的参与者，确保良好的用户体验。

## 指导和规范

### 指导

聊天参与者不应该仅仅只是问答机器人。当构建聊天参与者时，发挥创造力，使用已有的VS Code API创建使之与VS Code有丰富的融合。用户也喜欢丰富便利的互动，比如回复中的按钮、菜单项等在聊天中直接把用户带到参与者的交互。思考现实生活情景中AI可以帮助用户的地方。

并不是每个扩展都要贡献一个聊天参与者。聊天中有太多的聊天参与者可能会导致糟糕的用户体验。聊天参与者最好在您希望控制全部提示词，包括语言模型的说明时出现。您可以复用精细制作的Copilot系统消息，贡献上下文给参与者。

例如，语言扩展（比如C++扩展）可以通过各种其他方式贡献：

- 贡献将语言服务更智能的工具给用户使用。例如，C++扩展可以解析`#cpp`工具为工作空间的C++状态。这提供给Copilot语言模型正确的C++上下文以改善Copilot对C++的回答质量。
- 贡献使用语言模型的智能操作，可以与传统语言服务知识结合，提供更好的用户体验。例如，C++可能已经提供了“概括方法”的使用语言模型的智能操作，能够为新的方法生成合适的默认名称。

如果聊天扩展正在进行高成本的操作或者要编辑删除一些不能这么做的东西，应该清晰地询问用户许可。要拥有更好的用户体验，我们不鼓励扩展贡献多个聊天参与者。每个扩展最多一个聊天参与者这个简单模型在UI中是合适的比例。

### 聊天参与者命名规范

| 属性 | 简介 | 命名规范 |
| - | - | - |
| `id` | 聊天参与者全局独特的标识符 | <ul><li>字符串</li><li>使用扩展名作为前缀，随后跟随独特的ID</li><li>示例：`chat-sample.cat`、`code-visualizer.code-visualizer-participant`</li></ul> |
| `name` | 聊天参与者的名称，通过@符号由用户引用 | <ul><li>由字母、数字、下划线或连字符组成的字符串值</li><li>推荐仅使用小写确保与聊天参与者一致</li><li>通过引用您的公司名或功能确保聊天参与者的目的从名字上看明显</li><li>一些参与者的名称被保留。如果您使用保留名，会显示完整的合格名称，包括扩展ID</li><li>示例：`vscode`、`terminal`、`code-visualizer`</li></ul> |

# 教程：使用聊天API构建一个编程老师聊天参与者

在本教程中，您将学习如何与GitHub Copilot聊天体验融合，创建一个Visual Studio Code扩展。您会使用聊天扩展API贡献一个聊天参与者。您的参与者将作为编程老师提供编程概念的解释与示例练习。

## 准备
您需要下面的工具和账号完成本教程：
- [Visual Studio Code](https://code.visualstudio.com/download)
- [GitHub Copilot](https://marketplace.visualstudio.com/items?itemName=GitHub.copilot-chat)
- [Node.js](https://nodejs.org/en/download/)

# 第一步：建立您的项目
首先，使用Yeoman花盆VS Code扩展生成器生成扩展项目。
```bash
npx --package yo --package generator-code -- yo code
```

选择下面的选项完成建立：

```Bash
# ? What type of extension do you want to create? New Extension (TypeScript)
# ? What's the name of your extension? Code Tutor

### Press <Enter> to choose default for all options below ###

# ? What's the identifier of your extension? code-tutor
# ? What's the description of your extension? 留空
# ? Initialize a git repository? Yes
# ? Bundle the source code with webpack? No
# ? Which package manager to use? npm

# ? Do you want to open the new folder with Visual Studio Code? Open with `code`
```
一旦扩展项目生成，您会用到两个文件：`extension.ts`和`package.json`，在[扩展解剖文档](/2.%20开始/2.%20扩展解剖.md)了解更多。下面是速览：

- `extension.ts`是扩展的主要入口，包含聊天参与者的逻辑。
- `package.json`包括扩展的元数据，比如参与者的名称和简介。

删除`extension.ts`中`activate()`方法的自动生成的代码。这是您会编写我们的聊天参与者的逻辑的地方。

## 第二部：注册聊天参与者

在`package.json`文件，如下替换自动生成的`contributes`段落：
```json
"contributes": {
    "chatParticipants": [
    {
        "id": "chat-tutorial.code-tutor",
        "fullName": "编程老师",
        "name": "tutor",
        "description": "我可以教您什么？",
        "isSticky": true
    }
    ]
}
```

这段代码注册了带有下面属性的聊天参与者：
- 独特ID`chat-tutorial.code-tutor`，将在代码中引用。
- 全名`编程老师`，会在参与者回复的标题区域显示。
- 名称`tutor`，在聊天视图中以`@tutor`的方式引用聊天参与者。
- 简介“我可以帮您什么”，会在聊天输入框中作为占位文本显示。

最后，设置`isSticky: true`会在用户开始与参与者互动后，自动在聊天输入框中预留参与者的名称。

## 第三步：生成提示词

现在参与者已注册，您可以开始为编程老师实现逻辑。在`extension.ts`文件，为请求定义一段提示词。

创建一段好的提示词是从参与者获得最佳回复的关键。审阅[这篇文章](https://platform.openai.com/docs/guides/prompt-engineering)获得提示词工程的提示。

您的编程老师应该通过引导学生理解概念而不是直接提供答案，来模拟真正的老师。另外，老师应该围绕主题，避免回答非编程问题。

考虑下面两段提示词。哪个更像给出具体行为？

1. 你是一个乐于助人的编程老师。你的工作是教用户关于一些概念的简单介绍和示例代码。

2. 你是一个乐于助人的编程老师。你的工作是教用户关于一些概念的简单介绍和示例代码。用一系列消息组成的引导性概述回复。不要直接给用户答案，但是可以引导他们自己发现答案。如果用户回答非编程问题，礼貌地拒绝回复。

第二段提示词更具体，给参与者更清晰的回复防线。在`extension.ts`文件中添加本段提示词

```typescript
const BASE_PROMPT =
    '你是一个乐于助人的编程老师。你的工作是教用户关于一些概念的简单介绍和示例代码。用一系列消息组成的引导性概述回复。不要直接给用户答案，但是可以引导他们自己发现答案。如果用户回答非编程问题，礼貌地拒绝回复。';
```

## 第四步：实现请求处理器

现在提示词被选中，您需要实现请求处理器。这是处理用户聊天请求的东西。您会定义请求处理器，实现逻辑并处理请求，返回给用户回复。

第一步，定义处理器：

```typescript
// 定义聊天处理器
const handler: vscode.ChatRequestHandler = async (

## 第七步：添加信息历史以提供更多上下文

Copilot聊天最大的价值之一是遍历多个消息获得最佳回复。为了如此，您想要在聊天请求中递交参与者的消息历史。您可以通过`context.history`访问。

您需要检索历史，并添加到`messages`数组。您需要在`request.prompt`被添加前做这件事。

```typescript
// 定义聊天处理器
const handler: vscode.ChatRequestHandler = async (
    request: vscode.ChatRequest,
    context: vscode.ChatContext,
    stream: vscode.ChatResponseStream,
    token: vscode.CancellationToken
) => {
    // 初始化提示词
    let prompt = BASE_PROMPT;

    // 使用提示词初始化消息数组
    const messages = [vscode.LanugageModelChatMessage.User(prompt)];

    // 获取之前全部的参与者消息
    const previousMessages = context.history.filter(
        h => h instanceof vscode.ChatResponseTurn
    );

    // 添加之前的消息到messages数组
    previousMessages.forEach(m => {
        let fullMessage = '';
        m.response.forEach(r => {
            const mdPart = r as vscode.ChatResponseMarkdownPart;
            fullMessage += mdPart.value.value;
        });
        messages.push(vscode.LanguageModelChatMessage.Assistance(fullMessage));
    });

    // 加入用户消息
    messages.push(vscode.LanguageModelChatMessage.User(request.prompt));

    // 发送请求
    const chatResponse = await request.model.sendRequest(messages, {}, token);

    // 流传输回复
    for await (const fragment of chatResponse.text) {
        stream.markdown(fragment);
    }

    return;
};
```

## 语言模型
语言模型API允许您在Visual Studio Code扩展中[使用语言模型](/9.%20参考/1.%20VS%20Code%20API.md)和融合AI驱动特性与自然语言处理。

您可以在不同类型的扩展中使用语言模型API。典型的用法是[聊天扩展](#聊天参与者API)，您可以使用语言模型解释用户的请求，帮助提供回答。但语言模型API的使用不局限于此场景。您可能会在[语言](/6.%20语言扩展/1.%20概览.md)或[调试器](/4.%20扩展指南/16.%20调试器扩展.md)扩展，或者自定义扩展中的一条[命令](/4.%20扩展指南/3.%20命令.md)或[任务](/4.%20扩展指南/14.%20任务提供器.md)一部分使用语言模型。例如，Rust扩展可能会使用语言模型提供默认名以改善重命名体验。

使用语言模型API的过程包括如下步骤：

1. 构建语言模型提示词

2. 发送语言模型请求

3. 解释回复

下面的段落提供更详细的说明，解释如何实现以上步骤。

如要开始，可以探索[聊天扩展示例](https://github.com/microsoft/vscode-extension-samples/tree/main/chat-sample)。

## 构建语言模型提示词
要与语言模型交互，扩展应该首先构建它们的提示词，随后发送给语言模型请求。您可以使用提示词为语言模型说明您使用模型的大概任务。提示词也可以定义用户消息解释的上下文。

在构建语言模型提示词时，语言模型API支持两种类型的消息：

- **用户**：用于提供说明和用户请求
- **助理**：用于添加先前语言模型回复的历史作为提示词上下文

**注意**：现在语言模型API不支持使用系统消息。

您可以用两种方式构建语言模型提示词：

- `LanguageModelChatMessage` - 提供一条或多条消息字符串创建提示词。如果您只是刚开始使用语言模型API，可能会用到。
- [`@vscode/prompt-tsx`](https://www.npmjs.com/package/@vscode/prompt-tsx) - 使用TSX语法声明提示词。

如果您想要更全面控制语言模型提示词的组成，可以使用`prompt-tsx`库。例如，此库可以基于语言模型上下文窗口大小，动态适应提示词长度。了解更多关于[`@vscode/prompt-tsx`](https://www.npmjs.com/package/@vscode/prompt-tsx)或探索[聊天扩展示例](https://github.com/microsoft/vscode-extension-samples/tree/main/chat-sample)开始。

要了解更多关于提示词工程，我们建议阅读OpenAI优秀的[提示词工程指南](https://platform.openai.com/docs/guides/prompt-engineering)

**提示**：充分利用丰富的VS Code扩展API，以获得最相关的上下文，并在提示词中包含之。例如，包含编辑器中的活跃文件内容。

### 使用LanguageModelChatMessage类
语言模型API提供`LanguageModelChatMessage`类代表并创建聊天消息。您可以使用`LanguageModelChatMessage.User`或`LanguageModelChatMessage.Assistant`方法相应地创建用户或助理消息。

在下面的例子中，第一条消息提供提示词的上下文：
- 语言在回复中的人设（本例中为一只猫）。
- 模型在生成回复时应当遵循的规则（本例中，用有趣的风格和猫的比喻解释计算机科学概念）

第二段消息提供来自用户的特定请求或说明。它决定了需要完成的任务，并给定了第一条消息提供的上下文。

```typescript
const craftedPrompt = [
    vscode.LanguageModelChatMessage.User(
        '你是一只猫！仔细思考，每一步都必须表现地像猫。你的工作是用类似猫的有趣的风格解释计算机科学概念，同时使用猫的比喻。一直用解释概念开始恢复。一直包括代码示例。'
    ),
    vscode.LanguageModelChatMessage.User('我需要理解递归')
];
```

## 发送语言模型请求

当您构建完成语言模型提示词后，首先用`[selectChatModels](/9.%20参考/1.%20VS%20Code%20API.md)`方法选择您想要使用的语言模型。这个方法返回符合条件的语言模型数组。如果您同时实现了聊天参与者，我们建议您用聊天请求处理器`request`对象中传递的模型。这可以确保您的扩展不会干扰用户在聊天模型下拉框中选择的模型。随后，使用[`sendRequest`](/9.%20参考/1.%20VS%20Code%20API.md)方法发送请求给语言模型。

要选择语言模型，您可以指定下面的属性：`vendor`、`family`和`version`。使用这些属性搜索所有匹配给定供应者和模族，或通过ID选择指定模型。在[API参考](/9.%20参考/1.%20VS%20Code%20API.md)中了解更多关于这些属性。
 
**注意**：目前，语言模型模族支持`gpt-4o`、`gpt-4o-mini`、`o1`、`o1-mini`、`claude-3.5-sonnet`。如果您不确定使用什么模型，结合优秀的表现和品质，我们推荐`gpt-4o`。考虑到在编辑器直接互动的表现，我们推荐`gpt-4o-mini`。

如果没有模型匹配指定条件，`selectChatModels`方法返回空数组。您的扩展必须正确处理此情况。

下面的例子展示了如何选择所有模族、版本的`Copilot`模型：

```typescript
const models = await vscode.lm.selectChatModels({
    vendor: 'copilot'
});

//无模型可用
if(models.length === 0) {
    //TODO: 处理没有模型可用的情况
}
```

**重点**：在扩展使用前，Copilot语言模型需要用户许可。许可通过授权对话框实现。正因如此，`selectChatModels`应该作为用户初始化操作的部分调用，比如命令。

在选择模型后，您可以通过让模型实例调用`sendRequest`方法发送请求给语言模型。传递先前制作的[提示词](#构建语言模型提示词)和所有额外选项以及取消令牌。

在给语言模型API创建请求时，请求可能失败。例如，因为模型不存在，或者用户没有给语言模型API许可，或因为超出次数限制。使用`LanguageModelError`区分不同的错误类型。

下面的代码片段展示了如何发送语言模型请求：

```typescript
try {
    const [model] = await vscode.lm.selectChatModels({ vendor: 'copilot', family: `gpt-4o`});
    const request = model.sendRequest(craftedPrompt, {}, token);
} catch (err) {
    // 发送请求可能失败，因为
    // - 模型不存在
    // - 用户未许可
    // - 达到次数限制
    if (err instanceof vscode.LanguageModelError) {
        console.log(err.message, err.code, err.cause);
        if(err.cause instanceof Error && err.cause.message.includes('off_topic')) {
            stream.markdown(
                vscode.l10n.t("抱歉，我只能解释计算机科学概念")
            );
        }
    } else {
        // 添加其他错误处理逻辑
        throw err;
    }
}
```

## 解释回复

发送请求后，您必须处理来自语言模型API的请求。基于您的使用情景，您可以直接传递回复给用户，或解释回复，实现额外逻辑。

来自语言模型API的回复（[`LanguageModelChatResponse`](/9.%20参考/1.%20VS%20Code%20API.md)）基于流，使您能提供流畅的用户体验。例如：与[聊天API](#聊天参与者api)结合，持续回报结果和进度。

在处理流式回复时可能会出现错误，比如网络连接问题。确保在代码中添加合适的错误处理，来处理这些错误。

下面的代码片段展示了扩展如何注册一个命令，使之使用语言模型在活跃编辑器中使用有趣的猫名更改所有变量名。注意要追求更舒适的用户体验，让扩展流式传输代码返回给编辑器。

```typescript
vscode.commands.registerTextEditorCommand(
    'cat.namesInEditor',
    async (textEditor: vscode.TextEditor) => {
        // 用猫名和猫词替代活跃编辑器中的所有变量

        const [model] = await vscode.lm.selectChatModels({
            vendor: 'copilot',
            family: 'gpt-4o'
        });
        let chatResponse: vscode.LanguageModelChatResponse | undefined;

        const text = textEditor.document.getText();

        const messages = [
            vscode.LanguageModelChatMessage
                .User(`你是一只猫！仔细思考，一步步模仿猫。你的工作是用有趣的猫变量名取代下面代码中的变量。要有创意。注意仅回复代码。不要使用Markdown！`),
            vscode.LanguageModelChatMessage.User(text)
        ];

        try {
            chatResponse = await model.sendRequest(
                messages,
                {},
                new vscode.CancellationTokenSource().token
            );
        } catch (err) {
            if (err instanceof vscode.LanguageModelError) {
                console.log(err.message, err.code, err.cause);
            } else {
                throw err;
            }
            return;
        }
        // 在插入新内容前清理编辑器内容
        await textEditor.edit(edit => {
            const start = new vscode.Position(0, 0);
            const end = new vscode.Positio(
                textEditor.document.lineCount - 1,
                textEditor.document.lineAt(textEditor.document.lineCount - 1).text.length
            );
            edit.delete(new vscode.Range(start, end));
        });

        try {
            // 在语言模型带来代码时，将代码流式传输入编辑器
            for await (const fragment of chatResponse.text) {
                await textEditor.edit(edit => {
                    const lastLine = textEditor.document.lineAt(textEditor.document.lineCount - 1);
                    const position = new vscode.Position(lastLine.lineNumber, lastLine.text.length);
                    edit.insert(position, fragment);
                });
            }
        } catch (err) {
            // 异步返回流传输可能会失败，比如网络中断，或服务端错误
            await textEditor.edit(edit => {
                const lastLine = textEditor.document.lineAt(textEditor.document.lineCount - 1);
                const postion = new vscode.Position(latsLine.lineNumber, lastLine.text.length);
                edit.insert(position, (<Error>err).message);
            });
        }
    }
);
```

## 考虑事项
### 模型可用
我们不能指望任何指定模型永远保持可用。当您引用一个语言模型时，确保采取“防御性”方式发送请求给语言模型。这意味着您应该精细处理您不能访问特定模型的情况。

### 选择合适的模型
扩展作者可以选择最合适的模型。出于其表现和品质，我们推荐使用`gpt-4o`。要获取所有可用模型的列表，您可以使用这个代码片段：

```typescript
const allModels = await vscode.lm.selectChatModels(MODEL_SELECTOR);
```
> **注意**
> 推荐的GPT-4o模型有6.4万令牌限制。`selectChatModels`调用返回的模型对象具有`maxInputTokens`属性显示令牌限制。这个限制在我们学习更多关于如何使用语言模型的内容时会扩充。

### 速度限制
扩展应该负责任地使用语言模型，并关注速度限制。在扩展如何使用语言模型、每个扩展正在发送对少请求和它们各自的限额会有什么影响等方面，VS Code对用户是透明的。

由于速度限制，扩展不应该使用语言模型API执行集成测试。VS Code内部使用了专门的非生产语言模型提供给模拟测试，我们正在思考如何提供一个可扩展的语言模型测试方案。

## 测试扩展
语言模型API提供的回复并不具备决定性，这意味这您也许会获得同一个问题的完全不同回复。在测试扩展时，这个行为很棘手。

扩展构建提示词和解释语言模型回复的部分具有决定性，因此可以不使用实际的语言模型进行单元测试。然而，与语言模型本身互动并获得回复是非决定性，并且难以测试。考虑用更模块化的方式设计扩展代码，确保可以对指定部分进行单元测试。

## 发布扩展

一旦完成了AI扩展，您可以发布到Visual Studio市场：

- 在发布到VS市场前，我们推荐您阅读[Microsoft AI工具和实践指南](https://www.microsoft.com/en-us/ai/tools-practices)。这个指南提供了专门开发和使用AI技术的最佳实践。

- 通过发布到VS市场，您的扩展遵守[GitHub Copilot扩展可接受开发和使用政策](https://docs.github.com/en/early-access/copilot/github-copilot-extensibility-platform-partnership-plugin-acceptable-development-and-use-policy)。

- 如果您的扩展已经贡献了其他不适用语言模型API的功能，我们推荐您不要在[扩展清单](/9.%20参考/4.%20扩展清单.md)引入对GitHub Copilot的依赖。这确保不使用GitHub Copilot的扩展用户就可以使用非语言模型功能，而不需要安装GitHub Copilot。确保在访问语言模型时有正确的错误处理。

- 如[发布扩展](/7.%20测试与发布/2.%20发布扩展.md)中上传到市场。
