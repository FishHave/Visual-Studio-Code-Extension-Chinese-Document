# 工作空间信任扩展指南

## 什么是工作空间信任

[工作空间信任](https://code.visualstudio.com/docs/editing/workspaces/workspace-trust)是在用户使用VS Code打开工作空间时应对意外代码执行的安全风险而生的特性。例如，想象一个语言扩展，为了提供功能，也许需要从当前加载的工作空间执行代码。在这个情景下，用户应该信任工作空间的内容是非恶意的。工作空间信任在VS Code中集中管理此决策，并提供[受限模式](https://code.visualstudio.com/docs/editing/workspaces/workspace-trust#_restricted-mode)以保护自动代码执行的侵袭，从而扩展作者不需要处理这些基本问题。VS Code为接入扩展提供快速的静态声明和API支持，而不需要在各个扩展重复编写代码。

## 接入
### 静态声明
在扩展的`package.json`中，VS Code支持下面这个新的`capabilities`属性`untrustedWorkspaces`：
```typescript
capabilities:
    untrustedWorkspaces:
    { supported: true } |
    { supported: false, description: string } |
    { supported: 'limited', description: string, restrictedConfigurations?: string[] }
```

在`supported`属性中，接受下面的值：
- `true` - 此扩展在受限模式下完全支持，因为它不需要工作空间信任来执行功能。在实际运行中会如常启用。
- `false` - 此扩展在受限模式下不受支持，因为它的功能需要工作空间信任。它在得到工作空间信任前会保持禁用。
- `'limited'` - 此扩展的一些特性在受限模式下受支持。信任敏感特性在得到工作空间信任前被禁用。扩展可以使用VS Code API隐藏或禁用这些特性。可以调整工作空间设置`restrictedConfigurations`属性自动信任。

`description`属性中写入为什么需要信任，以帮助用户理解什么功能会被禁用以及它们需要在给予或拒绝工作空间信任前检查什么。如果`supported`设置为`true`，将忽略此属性。

`description`属性应当被添加入`package.nls.json`并在`package.json`中引用，以提供本地化支持。

`restrictedConfigurations`属性需要配置设置ID组成的数组。在列入的设置中，扩展在不受信任的工作空间中处于受限模式，不会给予工作空间定义值。

## 如何支持受限模式

为了帮助扩展作者理解工作空间信任包含什么，和受限模式下什么类型的功能是安全的，这里有一系列可以考虑的问题。

### 我的扩展是否有主入口

如果扩展没有`main`入口（例如主题和语言语法），扩展不需要工作空间信任。扩展作者不需要对扩展采取任何行动，因为无论工作空间是否受信任，它们会独立持续地发挥功能。

### 我的扩展是否依赖于打开的工作空间中的文件来提供特性

这个可以指代类似于由工作空间及其中的代码设置的设置项。如果扩展从不使用任何来自工作空间的内容，它也许不需要信任。除非，看看其他问题。
### 我的扩展是否视工作空间的任何内容为代码

最常见的例子是使用项目工作空间依赖，比如存储在本地工作空间的Node.js模块。恶意工作空间可能会提交模块的修改版本。因此，这对用户和扩展来说是个安全隐患。另外，扩展可能依赖于控制扩展或其他模块行为的JavaScript或者其他配置文件。有许多典型例子，比如执行打开的代码文件来控制错误报告的输出。

### 我的扩展是否使用定义在工作空间中的、决定代码执行的设置

您的扩展可能使用设置值作为扩展执行的CLI的标记。如果这些设置被恶意工作空间覆盖，它们会被利用作对您的扩展的攻击介体。另一方面，如果设置值仅用作检查某个条件，那么它就不会是一个安全风险，也不需要工作空间信任。例如，扩展可能检查偏好的命令解释器设置值是`bash`还是`pwsh`，以此决定显示什么文档。下面的[配置（设置）](#配置（设置）)小节有对设置作了指导，帮助您找到扩展的最佳配置。

这些并不是需要工作空间信任的所有情况。在我们审视更多扩展的过程中，我们会更新此列表。在考虑工作空间信任的事情时，使用此列表，思考您的扩展是否正在做类似的事情。

### 加入我不对我的扩展作任何修改

如同上面所提及的，不贡献任何东西给`package.json`的扩展会被当做不支持工作空间信任。它会在受限模式下的工作空间禁用，用户将收到提示：由于工作空间信任，一些扩展不会运作。对用户来说，这个举措是最安全谨慎的。尽管这是默认的方式，作为扩展作者，最好的做法是设置合适的值，指明您已经做出努力保护用户和扩展抵抗恶意工作空间内容。

## 工作空间信任API
如同上面所介绍的，使用API的第一步是在`package.json`中添加静态声明。最简单的接入方式是在`supported`属性中使用`false`值。再一次说明，即使你不做任何事，这也是默认的行为，但是对用户来说，你做了深思熟虑的抉择是良好的信号。在这种情况下，您不需要对扩展做其他任何事。在给予工作空间信任前它不会被启用，信任后扩展会知道已经得到用户的同意。但是，如果您的扩展的仅有部分功能需要信任，这似乎不是最好的选择。

对于希望开放功能给工作空间信任的扩展来说，它们应该在`supported`属性使用`limited`值，同时VS Code提供下面的API：

```typescript
export namespace workspace {
    /**
     * 为真时，用户已经明确信任工作空间内容。
     */
    export const isTrusted: boolean;

    /**
     * 事件，当当前工作空间被信任时触发。
     */
    export const onDidGrantWorkspaceTrust: Event<void>;
}
```

使用`isTrusted`属性判定当前工作空间是否受信任，使用`onDidGrantWorkspaceTrust`事件监听工作空间被授予信任的时间。您可以使用此API阻塞特定的代码路径，并在工作空间受信任时运行任何必要的注册。

VS Code同样暴露了上下文键`isWorkspaceTrusted`以便如下文介绍的在时句中使用。

## 贡献点
### 命令、视图或者其他UI
当用户不信任工作空间时，它们会在受限模式下工作，使用受限的功能浏览代码。任何您希望在受限模式下禁用的特性应当对用户隐藏。这可以通过[时句上下文](/9.%20参考/6.%20时句上下文.md)和其上下文键`isWorkspaceTrusted`做到。即便命令不会在UI中显示，它仍然能被调用，所以在您的扩展代码中使用上面的API阻止其运行或注册。

### 配置（设置）

受限，您需要检查您的设置，判断是否需要考虑信任。如同上面介绍的，工作空间可能定义恶意设置的值，同时您的扩展会使用之。如果您认识到设置的脆弱，应当使在`supported`属性中用`limited`，并在`restrictedConfigurations`数组中列出设置ID。

在添加设置ID到`restrictedConfigurations`数组后，VS Code在受限模式下会仅返回用户定义的设置值。您的扩展不需要任何额外的代码改动处理设置。获得信任后，除了工作空间信任事件外，额外触发一个配置更改事件。

### 调试扩展

VS Code会在受限模式下阻止调试，调试扩展通常不需要信任，应当在`supported`属性中选择`true`。然而，如果您的扩展提供了额外功能、命令或者设置但不作为内建调试流的一部分，您应当使用`limited`并遵从上面的指南。

### 任务提供器

类似于调试，VS Code在受限模式下阻止运行任务。如果扩展提供了额外的功能、命令或设置但不作为内建任务流的一部分，您应当使用`limited`并遵从上面的指南。其他情况可以指定`supported: true`。
